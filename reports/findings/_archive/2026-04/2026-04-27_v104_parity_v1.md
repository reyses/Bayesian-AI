## Confirmed: timezone gap

NT8 CSV "Entry time" column is in **PDT (UTC-7)**, not UTC.

Verified by trace on 2026-03-23:
- NT8 trade 1: Long entry @ 00:09 LOCAL, price 23898.75
- Python trade 1: Long entry @ 07:08 UTC, price 23898.50
- Match: 00:09 PDT + 7h = 07:09 UTC ≈ 07:08 UTC ✓ (within 1 minute)
- Price differs by 1 tick (0.25)

The 1-tick price difference is a separate issue — could be NT8 using TRADE
prices vs ATLAS_NT8 using QUOTE prices, or 1-tick fill spread. Small effect
in dollars but may shift pivot timing if it accumulates.

## The 2× trade count discrepancy

Per-day comparison (raw counts on 2026-03-23):

| Time window | NT8 trades | Python trades |
|---|---:|---:|
| 07:00 - 09:00 UTC (PDT 00:00 - 02:00) | 3 (Long, Short, Long) | 5 (rapid alternations) |
| 09:00 - 11:00 UTC | 4 | ~8 |
| Full day | 53 | 88 |

Python is detecting MORE pivots than NT8 with the same R=30. Possible causes:

1. **Bar data differs slightly** — confirmed at least one 1-tick price gap
   between NT8 and ATLAS_NT8 bars. If this happens repeatedly, my zigzag
   sees retracements NT8 didn't.
2. **NT8 trading-hours template excludes some bars** — e.g., the daily 1h
   Globex break (21:00-22:00 UTC). My data correctly excludes that, but
   maybe NT8 also excludes additional ranges.
3. **Bar timestamp alignment** — my dumped bars have :59 sec timestamps;
   NT8's bar boundaries may have :00 sec timestamps. If my sim treats
   bars as starting/ending differently, pivot conditions could fire on
   different bars.
4. **Python sim has a subtle bug** in zigzag detection — although manual
   trace through the code matches v1.0.4 source line-for-line.

## Why this matters for "money in account" decisions

Without parity, every Python prediction has unknown drift from NT8 reality.
Today's v1.5-RC episode is the worst-case demonstration:

- Python predicted +$5,021 on 95-day backtest (R=50)
- NT8 actual: −$1,396 on 20-day backtest (R=30)

Both R-mismatch AND parity-gap contributed to that miss. With v1.0.4 parity
established, future Python sims become trustable. Without it, Python is
guessing.

## Decision criterion for "parity achieved"

Per-trade match within tolerance:
- ≥95% of NT8 trades have a Python trade within 1 bar (60s) of entry
- Per-trade entry price within ≤1 tick (0.25)
- Per-trade exit price within ≤1 tick (0.25)
- Total PnL within ±5% of NT8 (after commission)

We are NOT there yet. Currently:
- ~17% match within 60s (8/517 trades have |entry_dt_diff| ≤ 60s — but this
  was using greedy nearest-neighbor matching, so likely understates true match)
- Total PnL diverges by $1,309 (>200% of NT8 net)
