PURPOSE
-------
Build a standalone screening tool that identifies tradeable patterns
directly from raw price data — no training checkpoints, no templates.pkl,
no dependency on the training pipeline.

The goal: find WHERE in price/time the tradeable setups live, THEN layer
physics features on top for context. Not the other way around.

THE PRICE I-MR METHOD
---------------------

What is I-MR?
  Statistical Process Control (SPC) chart. Used in manufacturing to
  detect when a process goes out of control. We apply it to price.

I Chart (Individual):
  - Each data point = one 15m close price
  - Center line = mean(close) from warmup period
  - UCL_I = center + 2.66 * MR_bar  (E2 constant for n=2)
  - LCL_I = center - 2.66 * MR_bar
  - Points outside limits = price is in "special cause" territory

MR Chart (Moving Range) — SIGNED:
  - MR = close[t] - close[t-1]  (signed, not absolute)
  - Positive = price went up, Negative = price went down
  - Every sign change and magnitude change is visible
  - MR_bar = mean(|MR|) from warmup period
  - UCL_MR = 3.267 * MR_bar  (D4 constant for n=2 subgroup)
  - |MR| > UCL = regime transition (price behavior changed character)

Why signed MR matters:
  - Every change in the MR is a potential pattern
  - Sign flips show direction reversals
  - Magnitude shows intensity
  - Runs above/below zero show trend persistence
  - UCL breaks in either direction mark regime boundaries


REGIME DETECTION
----------------

How regimes are found:
  1. Calibrate control limits from warmup period (first N days)
  2. Walk through analysis window
  3. New regime starts when |MR| > UCL_MR
  4. Merge tiny regimes (< 8 bars) into neighbors
  5. Each regime gets: direction (price trend), mean price, volatility

What a regime represents:
  - A period where price behavior has a consistent character
  - Bounded by special cause events (big bar-to-bar jumps)
  - The natural segmentation of price action
  - Replaces the fake ADX quartile bins

Regime metadata:
  - regime_id: sequential integer
  - start_idx, end_idx: bar range
  - n_bars: duration
  - mean_price: average close in regime
  - volatility: mean |MR| within regime
  - direction: LONG if price trended up, SHORT if down
  - price_change: close[-1] - close[0] within regime


ORACLE MFE/MAE
--------------

For each analysis bar:
  - Look ahead 16 bars (4 hours at 15m)
  - max_up = max(future highs) - entry price
  - max_down = entry price - min(future lows)
  - Direction from REGIME (not z-score):
    LONG regime -> MFE = max_up, MAE = max_down
    SHORT regime -> MFE = max_down, MAE = max_up
  - Win = MFE > MAE

THE THREE QUESTIONS METHOD
--------------------------

The screening follows a structured sequence of three independent analyses.
Each question builds on the answer to the previous one.

ANALYSIS A: Can we explain the PRICE?
  Y = close[t], X = 193 fractal context features
  Method: correlation screening + stepwise OLS regression
  Answer (1-day): adj-R2 = 0.9519 (YES, 95.2%)
  Key factors: 30m ADX (r=0.90), 30m DMI_diff (r=0.88), 15m ADX (r=0.87)
  Conclusion: The fractal context describes WHERE price is at any moment.

ANALYSIS B: Can we explain the DIRECTION standalone?
  Y = sign(close[t+1] - close[t]), X = 193 features (+ price anchor)
  Method: same correlation screening + stepwise OLS
  Answer (1-day): adj-R2 = 0.0870 (NO, only 8.7%)
  Key finding: all top factors have NEGATIVE correlation (mean reversion)
  Conclusion: Direction cannot be explained independently. The model has no
  anchor — it doesn't know WHERE price is, so it can't determine which way
  it goes. Adding raw price as a feature didn't help because the relationship
  between price level and direction is non-linear.

ANALYSIS C: Can we derive direction FROM the price model?
  Insight: if we can predict price[t] and price[t+1] with 95% accuracy,
  then direction = sign(predicted[t+1] - predicted[t]).
  Method: use the fitted price model from A, predict consecutive bars,
  derive direction from the difference.
  Risk: residual noise (~25 pts) may overwhelm typical moves (~9 pts).
  This tests whether 95% price R2 is "good enough" for directional use.

WHY THIS SEQUENCE MATTERS:
  - A establishes what the context CAN explain (price level)
  - B proves direction needs an anchor (can't be explained in isolation)
  - C tests whether the price model IS the anchor (derived direction)
  - Each analysis stands alone — no cross-referencing, no unified signals
  - The order is important: you must answer A before B makes sense,
    and B must fail before C is motivated

ANALYSIS C RESULTS (1-day, 2026-02-28)
  Overall accuracy: 55.7% (49/88) — barely above chance
  SNR: 0.668 (poor — residual std=21.8 > typical move=14.6)

  BUT — confidence-gated accuracy:
    |predicted delta| > 0:   55.7% (88 samples)
    |predicted delta| > 5:   68.8% (64 samples)
    |predicted delta| > 10:  73.9% (46 samples)
    |predicted delta| > 20:  73.9% (23 samples)

  KEY FINDING: The price model DOES carry directional information, but only
  when it predicts a move larger than the noise floor. When predicted delta
  exceeds ~10 points, accuracy jumps to 74%. Below that, noise dominates.

  This is a NATURAL CONFIDENCE GATE — the model itself tells you when to
  trust its direction (large predicted delta) and when to sit out (small
  predicted delta). You don't need a separate direction model; you need
  a confidence threshold on the price model's consecutive predictions.

  Asymmetry: DOWN prediction (60.5%) more accurate than UP (52.0%).
  Consistent with Analysis B's mean-reversion pattern — the model is
  better at calling reversals from extended states than continuations.

CONCLUSIONS SO FAR
------------------

1. Price explanation WORKS at all scales (89-97% R²) but the useful
   information shifts to higher TFs as the window grows.

2. Direction standalone is DEAD at all scales (3-9% R²). This is not a
   data problem or a feature problem — direction is not linearly
   predictable from spatial context.

3. Derived direction from price model has a SWEET SPOT at ~1 week:
   - Too short (1 day): not enough samples, noisy (56% overall)
   - Sweet spot (1 week): 67% overall, 80% at >20 pt confidence gate
   - Degrading (1 month): 60% overall, 68% at >20 pt gate
   - Dead (1 year): 50% — pure coin flip, confidence gate fails

4. The confidence gate is real but ONLY works within the sweet spot.
   It does not rescue the model at scales where SNR < ~0.2.

5. A single global linear model cannot handle multi-regime data.
   The path forward must involve either:
   a) Rolling windows (fit model per ~1-week window, slide forward)
   b) Regime-conditional models (fit per regime, derive direction locally)
   c) Non-linear methods (trees, neural nets that partition input space)
   d) Relative features (not absolute price, but price-relative-to-regime)


CRITICAL BUG: LOOK-AHEAD BIAS IN HIGHER TF ALIGNMENT (2026-02-28)
------------------------------------------------------------------

DISCOVERY: All previous results had look-ahead bias.

The problem:
  When sitting at a 15m bar with open timestamp t=10:00, the code looked up
  the higher TF bar with searchsorted(timestamps, t, side='right') - 1.
  This finds the bar whose OPEN time <= t. But OHLCV timestamps are bar OPEN
  times, so the bar at t=10:00 on a 1h chart covers 10:00-11:00 and its
  close/high/low/volume aren't known until 11:00.

  Look-ahead per TF (for base 15m bar):
    30m bar at 10:00 → completes at 10:30 → 15 min look-ahead
    1h  bar at 10:00 → completes at 11:00 → 45 min look-ahead
    4h  bar at 8:00  → completes at 12:00 → up to 3h45m look-ahead
    1D  bar at open  → completes at close → up to ~23h look-ahead
    1W  bar at open  → completes Friday   → up to ~5 days look-ahead

  TFs SMALLER than base (5m, 3m, 2m, 1m, 30s, 15s) are fine — their bars
  complete within the 15m base bar window.

THE FIX:
  For TFs with bar duration > base TF duration: use N-1 (previous completed bar).
    searchsorted(timestamps, t, side='right') - 2  (instead of - 1)
  For TFs with bar duration <= base TF: keep - 1 (bar completes within base bar).

  This means for 15m base at 10:00:
    30m → uses bar at 9:30 (covers 9:30-10:00, COMPLETED)
    1h  → uses bar at 9:00 (covers 9:00-10:00, COMPLETED)
    4h  → uses bar at 4:00 (covers 4:00-8:00, COMPLETED)
    1D  → uses previous day's bar (COMPLETED)
    1W  → uses previous week's bar (COMPLETED)

REFERENCE LITERATURE

[1] Shi, Chang-Ting (2018). "Signal Pattern Recognition Based on
    Fractal Features and Machine Learning." Applied Sciences 8(8):1327.
    College of Computer Science & Technology, Harbin Engineering Univ.
    URL: https://www.mdpi.com/2076-3417/8/8/1327

    METHOD: Extract 5 fractal dimensions from 8 modulation signal types
    (2ASK, 4ASK, 2FSK, 4FSK, 8FSK, BPSK, 16QAM, 32QAM), then classify.

    5 FRACTAL DIMENSIONS:
    a) Box dimension — covers signal with boxes of decreasing size,
       measures how the count scales. Best OVERALL discriminator of
       all 8 signals across the full SNR range.
    b) Katz dimension — ratio of curve length to planar extent.
       Best at separating 4ASK, 16QAM, and 32QAM from each other.
    c) Higuchi dimension — measures complexity from k-subsampled
       subsequences. WORST discriminator in this study.
    d) Petrosian dimension — fast approximation from zero-crossings
       of the first derivative. Best at separating 2FSK vs 16QAM.
    e) Sevcik dimension — normalizes waveform to unit square, then
       measures curve length. Similar to Katz in discrimination power.

    KEY BEHAVIOR: At low SNR (noisy), all signals cluster together —
    fractal dimensions overlap. As SNR increases, dimensions SEPARATE
    into distinct bands per signal type. This separation IS the signal.
    (This is exactly our reference image — the ideal we're targeting.)

    CLASSIFIERS TESTED:
    - Back-Propagation neural network: stabilized at ~85%
    - Grey Relation Analysis: stabilized at ~85%
    - K-Nearest Neighbor: stabilized at ~85%
    - Random Forest: stabilized at 96% <-- BEST by far
    Random Forest wins because it captures non-linear interactions
    between the 5 fractal features that linear methods miss.

    OUR ANALOGY:
    Their 8 modulation types = our price shape segments
    Their SNR axis = our bars-from-entry axis
    Their 5 fractal dims = our curvature + future fractal features
    Their key finding (RF >> linear) = why our linear R2 is 1% but
    the actual pattern structure exists (as proven by coherence tests)

[2] DSP Shape Detection Algorithms
    Ref: dsp.stackexchange.com/questions/64722/typical-shape-detection-algorithms

    KEY METHODS FOR WAVEFORM SHAPE MATCHING:

    a) Dynamic Time Warping (DTW):
       - Aligns two time series that are similar but time-stretched
       - Handles shapes that move at different speeds
       - O(n*m) but constrained variants are O(n)
       - shapeDTW variant: pairs locally similar structures, avoids
         matching points with distinct neighborhood structures
       - BEST FIT for our problem: two price shapes may have same
         pattern but at different speeds (fast vs slow trend)

    b) Fourier Descriptors:
       - Decompose shape contour into frequency components
       - Low-frequency coefficients = gross shape, high-freq = detail
       - Scale/rotation/translation INVARIANT (just drop coeff 0)
       - Phase information provides more accurate shape description
         than amplitude alone (WARP system, IEEE TPAMI)
       - POTENTIAL USE: describe each segment's shape as a compact
         frequency fingerprint for fast library lookup

    c) Template Matching via Cross-Correlation:
       - Slide template over signal, measure correlation at each offset
       - High correlation = shape match
       - Fast via FFT (O(n log n))
       - POTENTIAL USE: once segment library is built, scan incoming
         price data for matching templates in real-time

    d) Moment Descriptors:
       - Statistical moments (mean, variance, skewness, kurtosis) of
         the shape signal
       - Simple, fast, rotation-invariant
       - Can combine with Fourier for richer description

    e) Fractal Dimension Features (ties back to Shi 2018):
       - Box/Katz/Higuchi/Petrosian/Sevcik as shape descriptors
       - Each captures different aspect of signal complexity
       - Combined into feature vector -> classifier

    2. Compute fractal dimensions per segment (Shi's 5 features)
    3. Cluster segments by fractal profile (not by time)
    4. Within each cluster: Random Forest for direction prediction
    5. For real-time matching: DTW or cross-correlation against library

PROPOSED: SEED-FUNCTION SHAPE MATCHING (next direction)
========================================================

Date: 2026-02-28
Status: PROPOSED — not yet implemented

CORE IDEA:
Instead of blind KMeans clustering (75 mystery clusters), start with
KNOWN mathematical function shapes as seeds. Fit each price segment
to the best-matching function. The function type tells you the SHAPE,
the fitted parameters tell you the MAGNITUDE and TIMING.

This separates FORM from SCALE:
  Form  = WHICH function (V-shape, ramp, exponential, etc.)
  Scale = HOW BIG (depth, slope, amplitude)
  Timing = WHERE the inflection/peak/trough occurs

SEED FUNCTION LIBRARY (candidate shapes):

  1. LINEAR RAMP: y = a*t
     - Monotonic trend (up if a>0, down if a<0)
     - Parameter a = slope = rate of price change
     - Simplest shape, the baseline

  2. V-SHAPE: y = a * |t - t0| + b
     - Reversal pattern: drops to min at t0, then recovers
     - Parameters: a=steepness, t0=inflection bar, b=depth
     - This is what C25 actually is (WR=98%)

  3. INVERTED V (TENT): y = -a * |t - t0| + b
     - Peak at t0, then drops
     - Parameters: same as V but inverted
     - Bull trap / failed breakout pattern

  4. EXPONENTIAL: y = a * (exp(b*t) - 1)
     - Accelerating move (momentum building)
     - a=direction, b=acceleration rate
     - Breakout / trend acceleration pattern

  5. LOGARITHMIC: y = a * log(1 + t)
     - Decelerating move (momentum fading)
     - Early fast move that flattens out
     - Exhaustion / trend maturation pattern

  6. SIGMOID (STEP): y = a / (1 + exp(-b*(t - t0)))
     - Sudden level shift at t0
     - a=magnitude of jump, b=sharpness, t0=when
     - News event / gap pattern

  7. SINUSOID: y = a * sin(w*t + phi)
     - Oscillation / mean reversion
     - a=amplitude, w=frequency, phi=phase
     - Choppy / range-bound pattern

  8. FLAT: y ≈ 0 (constant within noise)
     - No directional movement
     - Consolidation / compression

FITTING METHOD:
For each N-bar segment (delta from entry):
  1. Fit all 8 seed functions via least-squares (scipy.optimize.curve_fit)
  2. Pick the best fit by lowest residual (SSE or adj-R2)
  3. Record: function_type, parameters, residual, goodness-of-fit
  4. If no function fits well (residual > threshold), mark as "unknown"

OUTPUT PER SEGMENT:
  - shape_type: "V_SHAPE" / "RAMP_UP" / "SIGMOID" / etc.
  - params: {slope, depth, inflection_bar, amplitude, ...}
  - fit_r2: how well the seed explains the segment
  - residual_pattern: what the seed DOESN'T explain

WHY THIS IS BETTER THAN KMEANS:
  - Interpretable: "V-reversal at bar 3 with -15 tick depth" vs "cluster 25"
  - Signed: the function type IS the trajectory signature
  - Parameterized: magnitude comes from fitted params, not cluster stats
  - Composable: a segment could be "ramp + oscillation" (two seeds)
  - Generalizable: same seed library works at any timeframe/instrument
  - Predictive: if you recognize the shape type at bar 4 of 8, you
    know what bars 5-8 should look like (extrapolation)

WORKFLOW: SHAPE FIRST, THEN MAGNITUDE
  1. Fit segment to seed functions -> identify shape type
  2. Within each shape type, analyze parameter distributions
     (e.g., V-shapes: what distribution of depths? inflection bars?)
  3. Regress 16D physics features -> shape parameters
     (which physics features PREDICT the depth of a V-reversal?)
  4. This gives a causal chain:
     Physics state -> predicts shape type -> shape has known trajectory
     -> trajectory predicts direction + timing + magnitude

CONNECTION TO EXISTING WORK:
  - Analysis H proved delta segments cluster into recognizable shapes
  - The seed functions are the NAMED versions of what KMeans found
  - C25 (V-reversal) maps to seed #2
  - C51 (steady ramp) maps to seed #1
  - C34 (down drift) maps to seed #1 with negative slope

ANALYSIS J: ADAPTIVE R-SQUARED SUB-CLASSIFICATION
=============================================================================
Date: 2026-02-28
Dataset: ATLAS full year (12 months, ~15,800 segments)
Script: tools/waveform_standalone.py

GOAL: Within each shape type from Analysis I, split segments into
sub-types that are internally consistent. Target: shape R² >= 0.90
(later found natural ceiling at ~0.88).

METHOD (evolved through multiple iterations):

  1. ADAPTIVE KMEANS ON SHAPE-NORMALIZED DATA
     - Normalize each segment to [0, 1] (decouple shape from magnitude)
     - Run KMeans on normalized data for k = 2..48
     - Pick k with highest min(R²) across all clusters
     - R² = mean Pearson r² per segment vs centroid (shape agreement)
     - min_n = 2 (minimum segments per cluster)
     - n_init = 20 (robust centroid initialization)

  2. IQR QUALITY GATE (outlier detection)
     - Per cluster: compute RMSE of each segment vs raw centroid
     - Fence = Q3 + 1.0 * IQR (aggressive, catches magnitude outliers)
     - Segments above fence -> MISFIT bin

  3. MISFIT RECLASSIFICATION
     - Collect all misfits across clusters
     - Run second adaptive_split on misfit pool
     - Misfits that form coherent sub-groups -> reclassified
     - Remainder -> UNCLASSIFIED (plotted with red border)

  4. TARGETED BISECTION (Phase 2)
     - After global k sweep, find worst cluster
     - Bisect via KMeans(k=2) on normalized data
     - Accept only if both halves improve over original

CRITICAL DESIGN DECISIONS:

  Shape-normalized R² vs raw R²:
    Raw R² requires shape AND magnitude agreement. Two segments both
    going up but by different amounts create huge residuals. Solution:
    normalize to [0,1] for R² computation, cluster on normalized data,
    but return raw centroids for plotting/analysis.

  IQR vs Pearson r gate:
    Tried Pearson r correlation gate first — found ZERO misfits because
    shape-normalized clustering already groups by shape. The problem was
    MAGNITUDE outliers within each shape cluster. IQR on raw-tick RMSE
    catches these correctly.

  Mean Pearson r² vs global R²:
    Global R² (SSres/SStot) is dominated by the largest segments.
    Mean per-segment Pearson r² weights all segments equally and measures
    shape fidelity independently of magnitude.

RESULTS (full year):
  - 19 shapes, adaptive k per shape (typically 3-12 sub-types)
  - Shape R² ceiling: ~0.88-0.89 (irreducible bar-to-bar noise)
  - R² target 0.90 not achieved for most shapes, but 0.80 achieved
    for all after adaptive splitting
  - IQR gate filters 5-15% of segments per shape (magnitude outliers)
  - Reclassification recovers ~30% of misfits into coherent sub-groups

  Summary table (representative shapes, full year):
    Shape               k   rcl  filt  minR²  status
    LINEAR_UP          12   +3    47   0.88   BEST=0.88
    LINEAR_DOWN        11   +2    39   0.87   BEST=0.87
    EXPONENTIAL_UP      8   +1    22   0.89   BEST=0.89
    V_SHAPE             6   +2    15   0.88   BEST=0.88
    RAMP_UP             9   +1    18   0.87   BEST=0.87
    FLAT                4   +0     8   0.89   BEST=0.89

  (k = clean sub-types, rcl = reclassified sub-types, filt = filtered)

R² CEILING INVESTIGATION:
  Tried: recursive bisection, iterative KMeans to k=48, targeted
  bisection, mean Pearson r², aggressive IQR (k=1.0), min_n=2.
  All converge to 0.88-0.89. This is NOT a methodology failure —
  it's the irreducible noise in 16-bar market segments. Each bar has
  genuine micro-randomness that no clustering can eliminate.

PLOTS:
  - Per-shape sub-type panels (0l_seed_classification.png updated)
  - Orange-bordered panels = reclassified from misfit pool
  - Red-bordered panels = UNCLASSIFIED (failed both passes)

KEY INSIGHT:
  The 0.88 R² ceiling tells us something fundamental: ~12% of within-
  shape variance is genuine noise. This matches SPC theory — common
  cause variation that no amount of sub-grouping can explain. The
  sub-type library is mature at this point; further splitting would

ANALYSIS S: EXIT TREND GUARD — BAND CONFLICT STUDY (2026-03-07)

CONTEXT:
  Root cause investigation found that fast TF bands hitting local resistance
  trigger band_tighten during trends that slow TFs still support. This causes
  too-early exits (207 trades captured <20% of oracle move). Fix #2 proposes
  suppressing tighten when band confluence direction matches trade side.

  See: reports/findings/2026-03-07_brain_aggregation.md (Fix #2)
  Code: core/timeframe_belief_network.py:973-981

METHOD:
  Simulate multi-scale z-scores from raw ATLAS price data:
    slow = 240-bar rolling regression (proxies 4h at 15m base)
    mid  = 60-bar rolling regression  (proxies 1h)
    fast = 15-bar rolling regression   (proxies 15m)
  For each oracle trade, hold 60 bars and track when fast TF hits resistance
  while slow TF still supports the trade direction.
  "With guard": suppress tighten when slow supports trade.
  "Without guard": every fast resistance hit = tighten.

RESULTS (7 days, 460 bars, 11 regimes):
  Total tighten events (no guard):   9,671
  Total tighten events (with guard): 5,641
  Suppressed by trend guard:         4,030 (41.7%)

  First tighten timing:
    No guard:   median bar 2, mean 7.6
    With guard: median bar 6, mean 15.0
    Delay:      +7.4 bars average (trades held longer)

  By direction:
    LONG  (176 trades): 45.6% tightens suppressed, avg MFE=107.2
    SHORT (284 trades): 39.1% tightens suppressed, avg MFE=114.0

  MFE by suppression rate:
    High suppression (>50% blocked): 232 trades, avg MFE=73.2
    Low suppression  (<=50%):        219 trades, avg MFE=155.5
    Difference: -82.3 (guard suppresses more on LOW-MFE trades)

INTERPRETATION:
  1. The guard is ACTIVE: 41.7% of tighten signals get suppressed. This is
     substantial — almost half of fast-TF resistance hits happen while the
     slow TF still supports the trade.
  2. First tighten delayed by 7.4 bars on average (from bar 2 to bar 6).
     This means trades are held 3x longer before first tighten.
  3. INTERESTING: high-suppression trades have LOWER MFE (73.2 vs 155.5).
     This suggests the guard correctly suppresses more on range-bound/
     low-opportunity trades where slow TF is neutral, while letting high-MFE
     trending trades keep their natural tightens.
  4. The guard is not blindly helping — it's filtering by context. Trades
     where slow TF confidently supports = few tightens suppressed (good:
     these are trending, high MFE). Trades where slow TF is ambiguous =
     many tightens suppressed (guard cautious, but these had less MFE anyway).

KEY INSIGHT #1: Interpolation/Extrapolation in 192D Space
  Analysis P isn't data leakage — it's proof that transitions through 192D
  space are deterministic. Two points define a vector; that vector resolves
  to a predictable outcome. The "leaked" B_current_MR just proved the
  concept — in practice, the transition signal comes from the trajectory.

KEY INSIGHT #2: 8-Point Trajectory Extrapolation (Octa-Point)
  Instead of predicting from 1 point (dead) or 2 points (needs future):
  - Use 8 SEQUENTIAL 192D snapshots to extrapolate the 9th
  - 1 point = position (R²=0.01)
  - 2 points = direction (98.8% but needs state B)
  - 8 points = full trajectory: position, velocity, acceleration, jerk, curvature
  - Can FIT the curve through 192D space and PROJECT where it goes next
  - This is time-series extrapolation in feature space, not price space

KEY INSIGHT #3: This IS the Bayesian Brain
  The brain already maps state → outcome distribution via lookup table.
  Current key: (template_id, direction) — a discrete hash.
  NEW key: trajectory hash — discretize the 8-point path shape through 192D.
  - Brain has seen similar 8-point trajectories → knows where 9th landed
  - Same data structure, same accumulation logic, same lookup at query time
  - No new model needed, no k-NN at runtime, no regression coefficients
  - Just a better hash function for the table that already exists

KEY INSIGHT #4: Neural Network Layer (future)
  The 8×192D trajectory (1,536 features) is a natural sequence input for
  LSTM/Transformer architectures. The brain table serves as baseline:
  - Brain: "I've seen this exact trajectory → here's the outcome"
  - NN:    "I haven't seen this exact one, but it's similar → interpolated outcome"
  - If NN disagrees with brain on high-confidence trajectory, brain wins
  - NN only gets authority on novel trajectories where brain has low density

  Ratio=1 → random walk (no edge). >1 → trending. <1 → mean-reverting.
  Replaces Hurst (ungrounded R/S statistic) AND ADX (double-smoothed ratio).

ADX: standard because Wilder published first (1978), not because it's best.
  5 layers: DM → smooth → DI → ratio → smooth → ADX. Variance ratio = 1 step.

  Individually weak (r=0.006), but DMI extreme + volume collapse = exhaustion signal.

REFRAME (user):

ENTRY 1 — conditional_probability_table.py (chop begets chop):
  Given K consecutive low-range legs, P(next leg's range class). After 3

  v1 CONFOUNDED (chop ratio co-grows with leg length). v2 fixed-early-window:
  chop measured over the first K tight legs only. Early-choppy wide legs run

  v1 CONFOUNDED — the chop table alternated 65/37/64/36/65 by K parity (a

  slightly wrong, but the edge is too small for a direction gate to fix
  profitably (cf. B1-B6 augmentation already tested at -$76/day).

ENTRY 4 — leg_age_hazard.py (is the move expended):
  Conditional-survival curve: given a leg has run M min, P(it continues
  +X min). HUMP-SHAPED hazard — the leg-death rate climbs 1% -> 9%/min over
  the first ~5 min, peaks at the ~5-min danger window, then DECLINES; a leg
  that clears ~5 min becomes MORE persistent. NOT monotone exhaustion. All
  four shape arms significant IS+OOS. Caveat: the steep early arm is partly
  the min_bars=36 (3-min minimum zigzag leg) construction floor; above the
  floor continuation RISES with age (mild momentum). A pure "old leg =
  cut it" time-stop is unsupported; only a danger-window-aware check is.

  INVERTS it (corr +0.34 -> -0.13): live, multi-scale agreement is a weak
  leg-MATURITY signal (high consensus = late leg = worse), not an edge.

METHOD LESSON — confounds in zigzag conditional tables (5 caught this arc):
  (1) Never measure DIRECTION on leg-anchored windows — zigzag alternation

ZIGZAG COLD-START DIVERGENCE — mid-day anchor vs beginning-of-time (2026-05-21)
=============================================================================

QUESTION
--------
Production (build_zigzag_pivot_dataset.py) runs detect_swings on each day's
full 5s closes, anchored at the DAY'S FIRST BAR. A live engine launched
mid-session cold-starts its zigzag at a mid-day anchor; a forward pass
anchors at the day's first bar. They diverge -- do they diverge ENOUGH to be
significant? (User reframe: the zigzag as a "beginning-of-time-dependent
feature, the worst kind.")

METHOD
------
tools/zigzag_coldstart_divergence.py. Per day, per offset K in {10,25,50,75}%
of the day: reference = detect_swings(closes[0:]); coldstart =
detect_swings(closes[K:]). resync lag = bars from K until the two pivot
sequences become an identical tail (detect_swings is deterministic -> once
they share a pivot + matching suffix they are identical forever after). Also
transient mis-tiled legs + leg-amplitude $ divergence. Production params
(ATR(14)x4, min_bars=36). 54 days, 216 cold-start samples.

RESULTS
-------
  never-resync    0 / 216 (0%)  -- every cold-start re-converges
  resync lag      median 27 min, mean 44 min CI [38,50], max 269 min
                  per-offset median: 10%->39, 25%->22, 50%->20, 75%->47 min
  mis-tiled legs  median 0, mean 0.7, max 4
  $ proxy         mean $103, median $80, CI [$91,$114], max $514
                  (a $ measure of mis-tiled leg-amplitude, NOT a P&L loss)

VERDICT: BOUNDED TRANSIENT -- not significant.
The zigzag is NOT a beginning-of-time-dependent feature; it is a
bounded-lookback feature with ~27 min median memory. The 0% never-resync rate
proves there is no permanent path-dependence -- the running extreme is
memoryless beyond the last pivot, so two runs collapse to identical the
instant they catch a common pivot. Median 0 divergent legs -> the transient
is mostly warmup dead-air, not active disagreement.

----------------------------------------------------
Prior-day zigzag priming = REJECTED (the forward pass runs detect_swings

2026-05-22  TRADE OUTCOME PROBABILITY TABLE -- continuation + recovery
================================================================================

User request: a probability table over all trades for entry-to-close P&L, MAE,
MFE. Sharpened to two CONDITIONAL questions: (1) given a trade is up +x, how
likely to close beyond +$200? (2) given a trade is down -d, how likely to
"work out in the end" (close net-positive)? A diagnostic asset -- has NOT
changed production code.

TOOL -- tools/trade_outcome_probability_table.py
  Per hardened leg, reconstructs the full entry->exit MAE/MFE from the 5s path
  (the legs CSVs carry only entry/exit/pnl). entry-to-close = CSV pnl (net of
  $6/leg friction). Key identity: "the trade ever reached open-PnL x" == MFE>=x
  (running maximum), so the conditional tables are simple MFE/MAE threshold
  conditionals -- no per-bar path needed. Emits IS and OOS separately on the
  R-multiple AND tick axes: (A) marginal survival CDF + histogram PDF; (B)
  continuation table, rows MFE>=x; (C) recovery table, rows MAE>=d; (D) joint
  2D MFE x MAE grid. Every conditional cell carries n + 95% bootstrap CI.

FINDINGS  (17,767 IS legs / 275 days; 2,936 OOS legs / 51 days)
  CONTINUATION -- P(close>=$200 | MFE>=x) rises 2.4% (x=0) -> ~25% (x~4R) ->
  ~85% (x~8R), IS and OOS tracking closely. A big close requires a big peak;
  "currently green" carries little information beyond how high it has run.
  STRUCTURAL ~1R GIVEBACK confirmed -- P(close>=x) holds (the leg ends at or
  above the level it reached) is flat ~44-50% across all x: once a leg reaches
  +x it ends at-or-above +x only ~half the time. close ~= MFE - R.

  RECOVERY -- the honest answer to "does going negative work out" is NO in
  expectation. P(close>0 | MAE>=d) falls 42.7% -> 25.6% by d=0.75R IS
  (41.5% -> 21% OOS), then survivorship-rebounds to ~38% at d>=1.25R (deep-
  drawdown legs still open are early entries into big moves). BUT mean close
  is NEGATIVE for every drawdown bucket d>=0.5R in both IS and OOS -- a leg
  that has drawn down >=0.5R is a net loser on average even though it
  sometimes recovers. The deep-d rebound is survivorship, not a hold signal.
  Re-confirms drawdown_stop_analysis: "hold the loser, it'll come back" is
  not supported.

  JOINT 2D -- P(close>0) is almost entirely MFE-driven: ~0% at MFE<1R, ~95%+
  at MFE>=2R; MAE only erodes it at the extreme corner. Ordering diagnostic:
  99-100% of recovered legs dipped before they rallied (mechanical).

Reports: reports/findings/trade_outcome_table/2026-05-22_trade_outcome_
probability_table.md  (+ reusable per_leg_excursions_{IS,OOS}.parquet)
Journal: docs/daily/2026-05-22.md

GLOBAL TWO-PHASE ARCHITECTURE & PCIE THRASHING (2026-06-08)
===========================================================

Catastrophic Bottleneck Discovery:
- Attempted to process the entire year with a ThreadPoolExecutor in Phase 1 (`run_year.py`) to parallelize PyTorch FISTA across multiple days.
- Result: Severe PCIe bus thrashing and GPU VRAM collision.
- Processing times skyrocketed from an expected ~10 minutes per day up to 50+ minutes for single volatile days (e.g., 2025-08-14 which had 176 data segments).

Architectural Pivot:
- PyTorch Sequential Gating: Phase 1 (`run_year.py`) was restructured to enforce strict sequential execution (one day at a time) for PyTorch operations, completely eliminating the GPU collisions.
- Joblib CPU Re-Injection: Restored `ElasticNetCV` parallel processing within `stage1_speed_pass.py` using `n_jobs=get_safe_n_jobs()`. The CPU now parallelizes internally *within* the single active PyTorch day. 
- Phase 2 Decoupling: Phase 2 chaos recovery was fully decoupled and is now executed globally at the very end of Phase 1 via a `ProcessPoolExecutor` that spans the entire dataset using purely CPU resources.

Results:
- This hybrid (Sequential GPU + Parallel CPU) completely bypassed the bottleneck.
- Average processing speed dropped back to an incredibly fast ~3 minutes per day without any deadlock or resource thrashing.
- Also patched the telemetry GUI windows to launch silently on Windows using `subprocess.CREATE_NO_WINDOW`.
