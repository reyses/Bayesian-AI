# 2026-03-13 — Regime Trading Framework (The Anchor)

## Session Summary
Fundamental architectural shift from scalping to regime trading. Identified the core problem:
system averages $1.67/trade (scalps) when market offers $22/trade with $10 risk (regime moves).

## Key Findings

### OOS Cold Start Investigation
- NOT a cold start issue. Single catastrophic day (Jan 14: -$136) caused by survival_stop
  allowing a -$111 exit. Fixed with breakeven gate in `core/exits/survival_stop.py`.

- 9,870 directional segments worth $30+ per month
- 76% have ZERO drawdown (price moves immediately in trade direction)
- With $10 SL (20 ticks): 94.9% win rate, $22 avg reward, 1:2.2 R:R

### DMI Crossover Validation (full IS, 12 months)
- 31,761 crossovers (50/50 LONG/SHORT)
- **46.3% WR raw** — barely better than coin flip
- $2.41 EV/trade, $76,696 total net
- Best filter: DMI spread < 2 → 53.3% WR, $4.68/trade
- ADX < 15 → 52.1% WR (low ADX crossovers are cleaner)
- **Conclusion**: DMI alone is insufficient. Needs 192D noise filter.

### Equity Simulation (10 MNQ cap = 1 NQ)
- 51,222 L2 segments, 87.1% WR with $10 SL
- Flat $10: $2.4M final equity, $0 max DD
- Dynamic 10%: $24M, $900 max DD, 10x multiplier
- Hits 10-contract cap by trade #24 (first trading day)
- All fractions converge due to cap

### I-MR Regime Segmentation (July 14, 2025)
- 14 regimes detected from MR UCL breaks on 1m data
- 11 profitable ($30+ MFE), avg $92 MFE, avg $27 MAE
- **Only 9% survive $10 SL** — regimes are multi-hour moves with real pullbacks
- Best regimes: R3 (LONG, $206 MFE, $20 MAE, 10:1 R:R), R7 (LONG, $89 MFE, $6 MAE)

### Noise Band Analysis (NEW)
- Average 1s noise: 2.12 ticks ($1.06) — very low
- MAE is 10-20x noise → drawdowns are REAL pullbacks, not random jitter
- $10 SL correctly covers noise but not real pullbacks
- $20 SL would capture ~64% of profitable regimes
- **Key insight**: I-MR regimes define DIRECTION, not exact entry. Need pullback entry.

### Scalping vs Regime Problem
- Current system: 30+ trades/day, $1.67 avg, pattern-template micro-movements
- Market reality: 5-10 regime moves/day, $22+ avg with $10 risk
- The oracle labels micro-patterns when it should label regime transitions
- Signal capture rate: ~3% of available L2 segments

## Architecture Evolution (Session 2)

### Revised Understanding
1. **I-MR segments the market into regimes** — MR UCL breaks are the ground truth boundaries
2. **Regimes define direction**, not entry — you trade WITH the regime, not AT its start
3. **192D state vector at lookback** — 10 bars before regime start = the "setup"
4. **Noise bands calibrate SL** — SL should be noise-relative, not fixed

### Regime Labeler Tool
- Step-through interactive tool: view one regime at a time
- Blue zone = 10-bar lookback (setup context), Green/Red = regime move
- 1s PnL path panel shows exact drawdown/profit trajectory

### Equity-Based Risk Model
- `risk = max($10, equity * 10%)`
- 10 MNQ max (= 1 NQ) — realistic for retail
- Anti-fragile: losses shrink position, wins compound
- Need to recalibrate: $10 SL too tight for regime starts, $20 may be better

## Session 3 — Regime Labeler Bug Fix

### matplotlib Backend Conflict (FIXED)
- `regime_labeler.py` interactive window never appeared — `FigureCanvasAgg is non-interactive`
- Root cause: `tools/research/__init__.py` imports `plots.py` which calls `matplotlib.use('Agg')`
  at module level. Any `from tools.research.data import X` triggers the package init, overriding
  TkAgg back to Agg even if set beforehand.
- Fix: `plt.switch_backend('TkAgg')` AFTER all imports (line 36). `switch_backend()` works even
  after the backend is initialized, unlike `matplotlib.use()` which can't override an already-set backend.

- Realized halfway through: marking too many micro-bounces. Later trades got longer/cleaner.

### Key Insight: Segmentation Is the Bottleneck
- The 192D features + brain + TBN work fine for understanding market state
- The problem is upstream: **where to draw segment boundaries**
- K-Means = random centroids, I-MR = 6-hour blocks — both bad segmentation
- Manual marking captures 5x more MFE per trade than current system ($9.9 vs $1.67)

### [RESEARCH] Trade Type Classification — Scalp vs Swing
- **Core problem**: system doesn't know what type of trade it's in
- Scalping is fine — but only when the system KNOWS it's scalping
- No point taking a 1-tick scalp if the move is a 10+ tick regime
- **Velocity = ticks per minute** distinguishes trade types:
  - Scalp: >10 t/min, 1-3 bars, tight SL ($5), quick TP
  - Swing: 5-10 t/min, 5-30 bars, wider SL ($20), trail exit
- **I-MR role redefined**: not entry signal, but CONTEXT layer
  - Answers: "am I inside a larger directional move right now?"
  - If regime aligns with trade → swing mode (hold, trail, ride it)
  - If no regime / choppy → scalp mode (tight SL, fast TP)
- **Implementation path**: exit engine gets `trade_type` parameter
  - I-MR regime state feeds classification in real-time
  - Different SL/TP/exit profiles per trade type
  - Position sizing adjusts: smaller for scalps (high frequency), larger for swings

## Session 5 — Seed Oracle Trainer + First Results

### Seed Tagging
- Added `tag` field to seed JSON: "Swing" (43) vs "Scalp" (4) based on <5 min cutoff
- Scalp IDs: T15, T29, T38, T41

### Seed Oracle Trainer (`tools/seed_oracle_trainer.py`)
- Extracts 192D features (12 TF × 16D) at every 1m bar on seed day
- Labels bars as entry/non-entry from manual seed timestamps
- Trains L1-regularized logistic (entry), logistic (direction), Ridge (MFE)

### First Results (47 seeds, July 14, single day)
- **Entry classifier**: 60.7% balanced accuracy (weak, above chance)
  - 27/192 features non-zero (L1 sparsity working)
  - 79% recall but only 16% precision (too many false entries)
- **Direction classifier**: **75.3% balanced accuracy** — STRONG SIGNAL
  - 93% full-data accuracy (19 LONG, 24 SHORT)
  - Top feature: 1m PID (coef -1.12 → SHORT)
- **MFE regressor**: CV R² = -1.5 (garbage, needs more data)

### Top Entry Features (multi-TF signal confirmed)
| Rank | Feature | TF | Coef |
|------|---------|-----|------|
| 1 | log1p_mom | 1m | +0.618 |
| 2 | z_score | 30m | +0.531 |
| 3 | self_adx | 15s | +0.465 |
| 4 | entropy_normalized | 15s | +0.383 |
| 5 | self_hurst | 30m | +0.313 |

### Top Direction Features
| Rank | Feature | TF | Coef | Meaning |
|------|---------|-----|------|---------|
| 1 | self_pid | 1m | -1.121 | +SHORT |
| 2 | log1p_mom | 5m | -0.521 | +SHORT |
| 3 | self_dmi_diff | 1m | -0.476 | +SHORT |

### Key Insight: Multi-TF Entry Signal
- Entry signal comes from 3 TF layers: **30m macro** (z-score, hurst) + **1m middle** (momentum) + **15s micro** (ADX, entropy)
- Direction signal concentrated in **1m and 5m** (PID, momentum, DMI)
- Validates that multi-TF stacking adds value — single-TF would miss the macro context

### [RESEARCH] TF Stepping — Manual Feature Ablation
- **Idea**: Instead of using all 12 TFs at once, step from macro to micro:
  - Run 1: 1h only (16D) → baseline
  - Run 2: 1h + 30m (32D) → does 30m add signal?
  - Run 3: 1h + 30m + 15m (48D) → ...
  - Continue down to 15s (full 192D)
- **Purpose**: Find the minimum TF set that captures entry signal
  - If 1h+5m+1m suffices, skip 30s/15s (saves compute in live)
  - If micro TFs (15s) add nothing, they're noise
  - If they DO add signal, the entry detector needs sub-minute data
- **Implementation**: Add `--tf-from` flag to `seed_oracle_trainer.py`
  - Mask out TF columns above the cutoff
  - Report accuracy per step → ablation curve
- **Status**: NOT STARTED — need more seed days first (47 samples barely supports 192D)

### Key Bugs Fixed
- **TF bucket mismatch**: Seeds tagged `timeframe='1m'` → 1m bucket centroids → no matches at 15s forward pass. Fix: seeds use `timeframe='15s'`
- **Property scale mismatch**: 1m regression window (21 bars = 21 min) vs 15s (21 bars = 5.25 min) → different z/vel/mom scales. Fix: compute MarketState from 15s data

### Seed-as-Replacement Failed (4 trades / 210 days)
- 47 seeds → 4 templates → too few centroids → almost no forward pass matches
- Realization: seeds should GUIDE discovery, not replace it

### Seed-Guided Discovery (final architecture)
- `--seeds` now means: run normal discovery → filter by seed-derived quality thresholds → cluster filtered set
- `compute_seed_thresholds(seed_path)`: extracts P25 of duration, MFE, range from seeds
- `filter_manifest_by_thresholds(manifest, thresholds)`: keeps only patterns meeting seed quality profile
- Example thresholds (47-seed file): duration >= 3.5m, MFE >= 13 ticks, range >= 20 ticks

```

  → Phase 1 SKIPPED (manifest from seeds)
  → Phase 2: TF-bucketed K-Means clustering (all in 15s bucket)
  → Phase 3+: normal pipeline (brain, forward pass, etc.)
```

## Session 9 — OOS Warmup Bug Fix

### Bug 1: Bar Index Offset in batch_compute_states (FIXED)
- `core/statistical_field_engine.py` line 511: `for i in range(rp, n)` dropped first 21 bars
- States list started at bar_idx=21, but TBN workers indexed from position 0
- Result: every TBN worker read states 21 bars ahead of actual current bar
- For 1h worker (~23 bars/day): `range(21, 23)` = only 2 valid states per day!
- **Fix**: Changed `range(rp, n)` to `range(n)`. CUDA kernel already outputs safe defaults
  (z=0, v=0, m=0) for bars < rp. Now all bars have states and indices align correctly.

### Bug 2: No OOS Regression Warmup (FIXED)
- OOS forward pass started each monthly file cold — regression had no history
- Macro TF workers (1h, 30m) started with z=0 for first 21 resampled bars
- User reported: "OOS always starts with very bad trades and then improves"
- **Fix**: For OOS mode, load the last IS monthly file (~21 trading days) and prepend
  to the first OOS file's data before `batch_compute_states` and TBN `prepare_day`.
  Regression kernel gets full history from bar 0. Trading loop uses `_warmup_offset`
  to align tick_all() calls with the prepended data.
