## Critical Discovery: Feature Parity Gap

**Two paths, different results:**
1. `process_one_day` (fast/AtlasCache) — batch SFE once per TF, `searchsorted` for each bar
2. `compute_79d_from_aggregator` (incremental) — batch SFE per bar call

**Comparison on same day (Apr 12):**
- 1m bars: 119/119 exact match (bars are identical)
- Features: 7.5% exact match (90/91 features diverge)
- Root cause: SFE called once-per-day (fast) vs once-per-bar (incremental) produces different states

**NOT a lookahead bug** — verified with sequential bar-by-bar feeding.
**The SFE batch function resets between calls** — per-bar calls don't accumulate state the way a single-pass does.

### What This Means
- Live engine using `compute_79d_from_aggregator` will NOT match training features
- Must use the fast method (AtlasCache-style) for live too
- Load pre-built TF parquets into memory, append new bars, `searchsorted` for current state
- Single SFE pass per TF, reuse states — exact match with training
