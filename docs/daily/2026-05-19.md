## Architecture decisions (autonomous calls)

  1. **On-demand V2 features, NOT streaming-V2-everywhere.** Compute
     V2 only at three moments:
       - Session start (B10 day-mode from cross_day_features)
       - R-trigger fire (B7 entry sizing -> binary skip)
       - K=5 after fill (B9 CUT/HOLD)
     The bar-by-bar feature path stays V1 91D for the position
     ledger / exit logic (already validated, no change).

  2. **LiveFeatureEngineV2 subclasses LiveFeatureEngine.** Inherits
     all bar ingestion + per-TF aggregation + dedupe. Adds
     `get_v2_vector(ts) -> np.ndarray[N_FEATURES_V2]` that calls
     core_v2 SFE on cached per-TF bars.

  3. **L5Decider has same `evaluate(state) -> Batch` interface as
     BlendedEngine.** engine_v2's per-bar call site does not change.
     Decider internally tracks zigzag state and fires entries on
     R-trigger.

  4. **Cautious mode on B10 cap days:**
       - B7 skip threshold raised from pred_R >= 1.0 to pred_R >= 1.5
       - B9 CUT threshold raised from pred < -50 to pred < -25
     No skip-day, no daily loss cap, no SL tightening yet.

     production is median of 42 TRs. Switched to median + 5-day

  3. **PositionsView.is_flat attribute bug.** L5Decider used `has_any`
     which doesn't exist -- silently always-flat -> exits never fired
     -> mock only emitted LONG trades. Replaced with proper is_flat
     property access.

  4. **Same-bar flip mechanics.** OOS legs have exit_ts == next leg's
     entry_ts (R-trigger fires the leg boundary). Engine had to emit
     exit + entry in same batch, and engine_v2 had to bypass
     `can_enter` (which requires is_flat) when the same batch contains
     an exit. Added `build_entry_order(allow_flip=True)`.

  5. **--pivot-source=replay mode (NEW).** User suggested using the
     forward pass engine directly. Implemented: L5Context loads
     oos_hardened_legs_full.csv at startup; replay fires R-triggers at
     each leg's entry_ts (NOT pivot_ts -- pivot_ts is the extreme bar
     which is FUTURE knowledge). Production live still uses streaming
     pivot detector.

  6. **Zero-slip mock fills.** Mock_bridge filled at `_last_price`
     which races with replay loop -> 1-bar-late fills causing ~$3.50
     /trade phantom slippage. Added `requested_price` stamp on every
     engine PLACE_ORDER; mock honors it. Net result: fills land at
     EXACTLY the bar close where engine made the decision.

  7. **B7/B9 threshold sync.** L5Decider's initial constants
     (B7 1.0/1.5, B9 -50/-25) didn't match the IS-calibrated values
     from forward_pass_1contract.py (B7 1.90/2.10, B9 +5/+15). Synced.

## cross_day_features WIRED INTO ENGINE STARTUP (B10 auto-currency)

### Why
B10 (day-regime sizer) reads DATA/CROSS_DAY/cross_day_features.parquet.
Engine Step 1 (dump NT8 days) + Step 2 (build V2 features) keep BAR data
+ features current, but NOT cross_day_features (separate artifact:
needs VIX/DXY + event calendar). Without it, B10 silently falls to
NORMAL mode -- a scope cut by accident. User: "sounds like you need to
complete it before deploying."

### Change (live/engine_v2.py _step2_build)
  - At the START of Step 2 (before the V1-features early-return), L5
    mode now runs `tools/sourcing/build_cross_day_features.py` as a
    subprocess.
  - The build is OFFLINE + fast (~10-30s): it reads pre-fetched
    VIX/DXY parquets (DATA/CROSS_DAY/raw/) + calendar CSVs + the 1m
    bars that Step 1 just refreshed. No network call in the critical
    startup path.
  - Non-fatal: on failure, logs a warning and B10 falls back to NORMAL.
  - Gated on engine_mode=='l5' -- blended mode unaffected.
  - Placement matters: runs BEFORE the "features up to date" early
    return, so it executes every L5 startup regardless of V1-feature
    state. And it runs AFTER Step 1 (run() calls step1 then step2), so
    the 1m bars it reads include the just-dumped days.

### VIX/DXY fetch ALSO integrated (full chain auto-current)
  fetch_vix_dxy.py (yfinance network call) now runs in Step 2 too,
  immediately BEFORE build_cross_day_features. Safety rails:
    - 90s timeout -- a yfinance hang cannot stall startup
    - Non-fatal -- on failure the existing raw VIX/DXY parquets are
      reused (slightly stale; only affects low-importance B10 features).
      Worst case == the prior behaviour, so integration is pure upside.
    - L5-mode-gated.
  Full B10 data chain now auto-current on every L5 startup:
    Step 1   -> dump missing NT8 bar days
    Step 2a  -> fetch_vix_dxy   (network, 90s, non-fatal)
    Step 2b  -> build_cross_day_features (offline, non-fatal)
    Step 2   -> build V1 features
  Network-call-in-startup risk is bounded: tight timeout + non-fatal
  means the engine always proceeds; B10 just uses slightly older
  VIX/DXY if the fetch is unavailable.

### Known future cleanup (NOT tonight)
  fetch_vix_dxy.py has a hardcoded END='2026-06-01'. After June 2026
  the VIX/DXY window must be extended. Low priority -- flagged here so
  it is not forgotten.

## DASHBOARD CLOSE BUTTON + SKIP DISPLAY + NT8 COMPANION (2026-05-20)

### CLOSE button (dashboard_v2.py)
Explicit in-UI CLOSE button. Saves geometry, sets shared_state
['shutdown'] (the flag the engine trade loop polls), disables all
buttons, shows "closing -- engine flattening + saving...". The engine
runs its graceful shutdown then os._exit(0) (which closes the window).
20s fallback destroys the window if the engine never exits.

### Skipped-signal display
"Show me when a signal is detected but skipped." When the zigzag fires
an R-trigger but no entry is taken:
  - L5Decider._record_skip captures {direction, reason, pred_R, thr}.
    Reasons: b7_low_conviction (pred_R < threshold), v2_warmup, no_v2_getter.
  - engine_v2 pushes a 'SIGNAL_SKIP' GUI message.
  - dashboard_v2: skip_flags deque (parallel to prices) -> hollow amber
    triangle on the chart at the skipped bar; skip counter + last-skip
    detail ("SHORT b7_low_conviction (1.82<1.90)") in the L5 status block.

### NT8 BayesianCompanion 1.0.0-RC (docs/nt8/BayesianCompanion.cs, NEW)
"Make an NT8 companion so what I see in Python I see in NT8."

  Architecture -- file-based, display-only, ZERO logic duplication:
    - engine_v2._write_l5_overlay() writes live/state/l5_overlay.txt
      every bar (atomic tmp+replace): r_price, zigzag dir/extreme,
      R-trigger threshold, B10 mode, B7/B9, skip_count, position,
      day_pnl + SKIP|/TRADE| event lines for the current bar.
    - BayesianCompanion is an NT8 INDICATOR you drop on your MNQ chart.
      It POLLS that file (~1s) and draws:
        * amber dashed horizontal line  -> zigzag running extreme
        * cyan dashed horizontal line   -> R-trigger threshold
        * bottom-left text block        -> r_price/dir/mode/B7/B9/skips/pos
        * hollow amber triangle         -> skipped R-trigger
        * green/red arrow               -> trade entry; gold diamond -> exit
  The companion computes NOTHING -- Python is the brain, NT8 is the
  windshield. No zigzag/B7/B9 re-implementation -> no drift. Event
  markers anchor at the current bar (barsAgo=0); the ~1s poll vs 5s
  bar cadence means a fresh event lands on the current bar -- no
  timezone conversion needed.

  Status: RC, NOT deployed. docs/nt8/ is dev space; copying to the NT8
  Indicators folder needs explicit approval.
