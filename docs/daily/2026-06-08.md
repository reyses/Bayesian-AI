## Addendum — finished `test_fista_parity.py`, found + fixed a real bug in `core_v2/math/fista_gpu.py`

**Context:** to decide whether to unify stage1 (GPU FISTA) and stage2 (CPU sklearn)
ElasticNet solvers, I finished the parity test (it only printed an ElasticNet
Jaccard and never computed a Group-Lasso one; no real-data case existed).

**Rewrote `test_fista_parity.py`** to measure three things and write a dated report
to `reports/findings/<date>_fista_solver_parity.md`:
1. Synthetic ElasticNet parity (FISTA-CV vs sklearn-CV active-set Jaccard).
2. Synthetic Group Lasso parity (the missing Jaccard).
3. **Real segment-data parity** — replays stage1's exact path (screen → poly →
   solver → OLS → `categorize_segment`) with FISTA vs sklearn and reports the
   rate at which the chosen solver flips the final **tier** (the decision number).

**Decision on unification: don't.** On real data (day 2025_07_29, 8×120-bar blocks)
term-selection Jaccard was 0.762 but **tier disagreement was 0/8 (0%)** — the tier
(the only consumed output) is robust to the solver. Keep the doc note; skip the
code change. (Quick sample; `--n-blocks 25` across more days for firmer ground.)

**BUG FOUND — `elasticnet_fista_cv` CV step size (now FIXED).** Synthetic parity
exposed it: FISTA selected `alpha` exactly ~1000× sklearn's (= 1/eps), kept **0
features**, Jaccard 0.000. Root cause (confirmed by an instrumented sweep,
`c:/tmp/fista_diag.py`): the smooth part is `(1/2N)||Xz−y||² + (lam_l2/2)||z||²`
with gradient Lipschitz `L + lam_l2`, but the code used a scalar `step = 1/L`,
ignoring `lam_l2` (which enters via the `z*lam_l2` gradient term, not the Gram
diagonal — that's why the non-CV `elasticnet_fista`, which adds lam_l2 to the
diagonal *before* computing L, was never affected). High-alpha columns (large
lam_l2) overshot the stability bound → **diverged to NaN** → `torch.argmin` then
returned the NaN column (= alpha_max) → CV always "selected" max regularization →
all-zero final fit. **Data-dependent**: triggers when `alpha_max·(1−l1_ratio)` is
large vs the Gram norm L — i.e. strong-trend (PRISTINE-class) segments, exactly
what stage1 hunts. The 8 sampled real blocks didn't trigger it, but trending
blocks can.

**Fix (2 edits in `core_v2/math/fista_gpu.py`):**
(1) per-column step `1/(L + lam_l2 + 1e-6)` so every alpha respects its true
Lipschitz; (2) `nan_to_num(mean_mse, nan=+inf)` before `argmin` as defense in
depth. Blast radius contained — `elasticnet_fista_cv` is imported only by
`stage1_speed_pass.py` and the parity test; nothing else uses it.

**Verified:** post-fix synthetic ElasticNet Jaccard 0.000 → **0.981** (FISTA alpha
now equals sklearn's to 5 dp; active counts match), Group Lasso still 1.000,
real-data tier disagreement still 0/8. The parity test is now the regression guard.

**Impact / next run:** re-running stage1 will now correctly fit strong-trend
segments that previously collapsed to 0 terms → expect MORE PRISTINE / fewer false
chaos blocks on trending days, and stage1 tiers that better match stage2. Existing
artifacts on disk are stale w.r.t. this fix; regenerate when convenient. Still no
live trading path touched.

## Addendum 2 — Global Two-Phase Architecture & PCIe Thrashing Fix

**Context:** Later in the session, we attempted to run `run_year.py` using a parallel `ThreadPoolExecutor` to process multiple days of PyTorch FISTA simultaneously. 
This resulted in catastrophic PCIe bus thrashing and GPU VRAM collisions, causing processing times to balloon to 20-50+ minutes per day. 

**Fixes Applied:**
1. **PyTorch Sequential Gating:** Re-architected Phase 1 (`run_year.py`) to execute strictly sequentially (1 day at a time). PyTorch handles the massive polynomial interactions cleanly in isolation.
2. **CPU-Heavy Joblib Re-Injected:** Restored the dynamic RAM `n_jobs=get_safe_n_jobs()` logic inside `stage1_speed_pass.py`. While PyTorch runs sequentially, the CPU leverages 24+ Joblib workers to blast through the `ElasticNetCV` scan inside that specific day. 
3. **Phase 2 Global:** The Phase 2 Chaos recovery pool was fully decoupled. It now runs *only* after Phase 1 finishes the entire dataset, creating a massive global CPU `ProcessPoolExecutor` that crunches all chaos without touching the GPU.

**Results:**
The hybrid architecture completely resolved the bottleneck. The sequential PyTorch + parallel CPU joblib pipeline successfully crunches through Phase 1 at an average speed of **~3 minutes per day**, fully utilizing the hardware without deadlock or thrashing. 
The annoying blank CLI window spawned by `telemetry/gui.py` on Windows was also fixed via `subprocess.CREATE_NO_WINDOW`.
