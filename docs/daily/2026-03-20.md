## Key Discovery: F_momentum Parity Break
- Live F_momentum median = 12.3 vs ATLAS median = 218.5 (17x difference)
- Root cause: PID controller uses `np.cumsum(z_scores)` — integral term accumulates over full history
- ATLAS has 417K contiguous 1m bars. Live had ~2.5K bars (resampled from 15s)
- cumsum starts from zero in live → near-zero F_momentum → sensor gate blocks EVERYTHING
- Evidence: 303,897 sensor blocks in 2 hours of live trading. OOS took 25 trades in same window (+$549)
- Fix: pre-compute states from ATLAS, load at live startup

## ARCHITECTURE DECISION: Absolute vs Delta (CRITICAL)

**Every computation in MarketState is cumsum-based (absolute):**
- F_momentum = kp*error + ki*CUMSUM + kd*derror
- DMI = Wilder smoothing (exponential cumsum)
- ADX = double smoothed cumsum
- P_at_center = regression window (history-dependent)
- velocity = derivative of cumsum

**Consequence**: Starting from different bar counts produces different values
for the SAME price bar. This is the fundamental reason OOS != Live.

**Delta-based engine (long-term fix):**
- Measure bar-to-bar CHANGES, not absolute levels
- Delta is history-independent: +15 change = +15 whether baseline is 218 or 12
- Gates check "is F_momentum decreasing?" not "is F_momentum < -1.0"
- No warmup needed. Instant parity. No pre-computed states.

**Applies to**: crow brain design (use deltas as features, not absolute values),
CNN input (trajectory of changes over 10 bars), sensor gate thresholds
(z-score normalization already proposed in Fix 2).

**Action items**:
- Crow brain: feature vector should use bar-to-bar deltas
- CNN: input channels should be change sequences, not level sequences
- Fix 2 (sensor gate normalization): use rolling z-score of F_momentum instead of absolute
- Eventually: refactor StatisticalFieldEngine to output deltas natively
