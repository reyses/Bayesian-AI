# 2026-05-20 — Mid-Leg Research · First SIM Run · Parity/B9-Bug Fix · Zigzag-Only Refactor

## Context
The user asked whether the live engine can "late-join" an R-trigger it
missed — enter a leg mid-flight if a positive-P&L shot is still available.
Two forks were specified: **Fork 1** = gate the late-join with B9; **Fork 2**
= augment with the B1-B6 pivot-structure models ("those carry past state
since zigzag is dumb fire"). The user pointed at `examples/example1.png`
(NT8 chart, ~9 legs / 5 h) saying "I'm seeing genuine lost signals," then
went to sleep directing: "test both forks and other options." Autonomous run.

## Verdict — MID-LEG ENTRY: DO NOT BUILD
There is **no addressable population of missed legs**. A 1-contract greedy
engine already catches **2,922 / 2,926** OOS legs (misses 4 in 51 days, all
losers). Hardened zigzag legs are a *sequential partition* of the price path
— leg N exits where leg N+1 is born — so the engine is essentially never
"busy and missing a parallel signal."

## Experiments (all 51-day sealed OOS, metric-compliant)

**E1 — Fork 1: B9-gated late join, unconstrained.**
`tools/forward_pass_midleg_entry.py`. Per K, principled gate (B9 pred > $6
friction): K=5 **+$303/day CI [+160,+457] SIG**; K=10 **+$232 CI [+86,+388]
SIG**; K=30 +$78 (not sig); K=60 +$50 (not sig); K=120 −$19 (not sig).
Unconditional join loses at every K — the gate is essential. Signal decays
hard with lateness. **Nuance**: the baseline already enters every leg at
K=0; late-join at K=5 enters the *same* legs 25 s later (strictly worse).
The +$303 is the value of B9-*gating*, not new money — it re-measures the
B9 K=5 cut already in the live stack.

**E2 — Fork 2: B1-B6 augmentation.** `tools/midleg_b1to6_augmented.py`.
Model A (B9's 190 features) vs Model B (A + 18 B1-B6 prediction features),
both trained on full IS, sealed-OOS tested. delta(B−A): K=5 **−$76/day CI
[−158,−2] SIGNIFICANTLY HURTS**; K=10 −$79; K=30 −$46; K=60 −$94; K=120 +$6
— neutral-to-harmful everywhere. **Fork 2 rejected.** B1-B6 are genuinely
in-distribution mid-leg (trained on all 1m bars), so the user's "they carry
state" intuition was *conceptually* correct — but the B9 GBM already extracts
that pivot-structure from the raw V2 features; the 18 stacked B1-B6
predictions are deterministic functions of the same inputs and add only
noise. Same failure pattern as the rejected lead-in PCA. Standalone structure
gates (b1/b3/b5) lose money; adding a structure filter to B9 *removes*
profitable joins (B9 & b5-early = +$182 vs B9-alone +$303). Pullback filter:
non-significant nudge only.

**E3 — position-constrained 1-contract sim.** `tools/midleg_constrained_sim.py`.
Baseline greedy: +$122/day net of friction, 2,922 legs taken (57.3/day),
gross ≈ +$466 — matches the known FLAT-1c ~$454 → **sim calibrated.**
Late-join (B9-gated, any max_k): **incremental −$1/day, CI [−2,+1], not
significant; only 4 late-joins in 51 days.** The E1 ceiling does not survive
the position constraint.

**E4 — leg overlap structure (the mechanism).** `tools/midleg_overlap_analysis.py`.
**99.8% of consecutive-leg gaps are exactly 0**; engine utilisation **99.9%**;
only 4/2926 legs overlap any earlier leg. Leg duration median 655 s. The 4
missable legs are all losers (−$110, −$32, $0, −$46) — catching them loses
money.

## What the "lost signals" actually are
NOT busy-missed parallel legs. They are: (1) **cold-start** — ~7 first-of-
session legs/day lost to the ~20-min (240-bar) feature-engine + zigzag
warmup, when the engine cannot compute anything; (2) deliberate **B7 skips**
(the filter working as designed). Late-join cannot address either.

## Recommendation
1. **Mid-leg entry — SHELVED.** Structural (sequential legs), not tunable;
   will not improve with more data or contract scaling.
2. **Build zigzag-state priming** — replay `detect_swings` over prior-day
   history at startup + pre-warm the V2 feature engine, so the engine is
   never blind at session open. This is the real fix for the lost signals.
   Small live-engine change — **awaiting user approval.**
3. **SIM deployment unaffected** — L5 stack (B7+B9+B10) goes to SIM as-is.

# Afternoon/Evening — First L5 SIM run, parity investigation, B9-horizon bug, zigzag-only refactor

## First L5 SIM run — −$379, then explained as an engine bug
The L5 engine ran live SIM on 2026-05-20. User ended the run at ≈−$200 (the
`v2_trades` log spans ~13.6 h / ~29 trades, total −$379). User demanded a
**parity check** before interpreting it: "run the same period in live sim and
forward pass… if there's no parity it's back to the drawing board."

## Drawdown-stop research (user idea: "cut at −$100 unless evidence of recovery")
`tools/drawdown_stop_analysis.py`. Of OOS legs that hit −$100 drawdown, **76%
recover** above −$100. B9's AUC at the −$100 checkpoint = **0.465** (no
signal). Only 25 OOS / 152 IS legs hit −$100 — too few to train a specialised
model. Verdict: the "unless evidence" clause is unsatisfiable; a dumb −$100
stop costs ≈−$31/day [CI −55,−11] and cuts mostly recoverers. **Don't add.**

## Parity investigation — found a real engine bug
- **Price parity** (`tools/parity_live_vs_forward.py`): re-priced the 28 live
  trades at `requested_price` (zero-slip forward pass) = −$476; live = −$379;
  net slippage **+$96 favourable** (not a cost).
- **Decision parity**: ran the mock forward pass for 2026-05-20 → found the
  mock tool broken (B9 disabled), then the real bug.

**THE B9-HORIZON BUG.** `core/ledger.py:396` computes `bars_held =
(ts−entry_ts)//60` — elapsed **minutes**. `l5_decider` fired B9 at
`bars_held == B9_K` (=5) → **5 minutes** after entry. But the B9 model
`b9_*_K5.pkl` was trained for K=5 in 5-SECOND-bar units
(`build_trade_trajectory_dataset.py`: `bar_ts = entry_ts + K*5`) =
**25 seconds**. The live engine fired the 25-second B9 model at the 5-minute
mark — **12× too late, every trade.**

**The mock entry_ts bug.** In mock mode the position `entry_ts` was set to
wall-clock (from the mock fill) not the replayed bar-time → `bars_held` went
negative → B9 never fired in the mock at all.

## Fixes (verified)
- `live/l5_decider.py`: B9 now fires off a 5-second-bar counter — when
  `(_bar_count − entry_bar_count) >= B9_K` (5 bars = 25 s). `pos.bars_held`
  (minutes) no longer used for the B9 trigger. `core/ledger.py` NOT changed
  (blended engine still depends on `bars_held`=minutes).
- `live/mock_bridge.py`: fills now stamp `fill_time` with the replayed bar
  timestamp (`_last_bar_sent_ts`), not wall-clock.
- Both verified — diffs read; mock re-ran clean (`bars_held` positive, B9
  fires at 25 s, timestamps bar-time).

## Parity verdict
The validated strategy (B9 at 25 s), re-run on 2026-05-20 ≈ **−$59** (a flat
day). The buggy live engine did −$379. **The −$379 was the bug, not the
strategy.** Residual: live vs mock still tile legs differently (24 vs 28
trades) — root cause = the live engine **cold-starts** with no zigzag state
while the forward pass warm-starts. Fix = zigzag-state priming (Phase 2).
