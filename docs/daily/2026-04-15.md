## Critical findings (morning session)

The live session that ran overnight (2026-04-14 → 2026-04-15) ended with a
deadlocked engine and a phantom NT8 position. Investigation traced three
interacting bugs:

### Bug 1 — NT8 bridge had no scale-out path (PARITY BLOCKER)

`docs/NT8_BayesianBridge.cs:909` (pre-fix) hardcoded:

```csharp
side == "BUY" ? OrderAction.Buy : OrderAction.SellShort
```

Every `PLACE_ORDER SELL` was sent as `OrderAction.SellShort` — which is
"open new short position" in NT8's API. There was no `OrderAction.Sell`
code path. So every chain exit (`CHEXIT_S_*`) was opening a new opposing
short instead of scaling out the long. In a netted account, NT8 then
FIFO-matched the new SHORT against the oldest LONG (the primary entry,
not the chain), leaving the chain still open with nothing managing it.

The NT8 grid confirmed it exactly: `Sell short, 25977, CHEXIT_S_005`
where the engine intended a scale-out, not a new short.

### Bug 2 — OrderManager full-close branch on `_awaiting_exit_fill`

`live/order_manager.py:307` (pre-fix):
```python
if qty >= self.position.qty or was_awaiting_exit:
    # Full close
    exit_qty = self.position.qty
```

`build_scale_out_order` was setting `_awaiting_exit_fill=True`, so when
the FILL came back, `was_awaiting_exit=True` triggered the full-close
branch. PnL was computed against the avg of both contracts (-$102 instead
of the actual -$13.50 for the chain alone) and `position.qty` was reset
to 0, completely diverging from NT8's actual position.

### Bug 3 — Silent tier/direction churn in the engine

The engine's `_check_bar` could re-classify `direction` and `entry_tier`
mid-trade without going through a close+open boundary. Because
`entered = in_pos AND not prev_in_pos` is False if prev was already in
position, the wrapper missed every silent flip — no orders sent to NT8,
no events logged, but the ledger showed direction/tier changing.

The session log had **9 silent flips** between 23:00 and 05:45.

## Fixes shipped

### File 1: `live/protocol.py`
- Added `position_effect` parameter to `place_order()` (default `OPEN` for
  backward compat). Field travels in PLACE_ORDER message.

### File 2: `docs/NT8_BayesianBridge.cs` (v7.0.0)
- Reads `position_effect` from PLACE_ORDER. Maps:
  - `OPEN`   → `OrderAction.Buy` / `OrderAction.SellShort`
  - `REDUCE` → `OrderAction.Sell` / `OrderAction.BuyToCover`
- REJECTS REDUCE with no matching opposing position (instead of silently
  turning a "close my long" into "open a new short").
- Caps `qty` at current position size to prevent over-close.
- Bumped `BRIDGE_VERSION = "7.0.0"`.
- **DEPLOYED** to `~/OneDrive/Documents/NinjaTrader 8/bin/Custom/Indicators/`
  — user must recompile before next run.

### File 3: `live/order_manager.py` — handshake state machine
- New `OrderIntent` enum: `OPEN | REDUCE | FLIP`.
- New `OrderState` enum: `PENDING → SENT → ACKED → WORKING → FILLED`,
  plus terminal `CANCELLED | REJECTED | TIMED_OUT`.
- `OrderRecord` now tracks `intent`, `is_chain`, `expected_position_after`,
  and timestamps for each handshake stage.
- `build_scale_out_order` passes `position_effect=REDUCE`.
- `on_fill` uses `rec.intent` (not the ambient `_awaiting_exit_fill` flag)
  to classify scale-out vs full close. Falls back to position-direction
  inference for untracked orders (manual NT8 entries).
- After every fill, validates actual position against
  `rec.expected_position_after` and surfaces mismatch via
  `last_reconcile_error`.
- New `check_pending_timeouts()` watchdog flags orders stuck in
  SENT/ACKED/WORKING beyond their respective deadlines (5s/10s/30s).
- `is_awaiting_open()` / `is_awaiting_reduce()` derive from rec states
  rather than ambient flags. Legacy `_awaiting_*_fill` booleans kept as
  cached snapshots refreshed via `_refresh_pending_flags()`.
- `build_exit_order` pre-registers a `BAY_CLOSE` OrderRecord so on_fill
  for CLOSE_POSITION fills can find it.

### File 4: `live/engine_v2.py` — wiring + observability + shutdown
- Steps 4/5/6 (`_step4_sync`, `_step5_catchup`, `_step6_verify`) all
  check `shared_state.get('shutdown')` and abort cleanly. No more
  30-second-blocked dashboard close during startup.
- `run()` skips remaining steps if `_shutting_down` flips during 4/5/6.
- After every `await self._client.send(order_msg)`, the engine calls
  `self._orders.mark_sent(order_id)` to advance the rec to SENT state
  (starts the ACK timeout clock).
- Main loop watchdog: `check_pending_timeouts()` runs each iteration,
  pushes ALERT events to the dashboard for any timed-out order, and
  triggers a position resync.
- Main loop also surfaces any pending `last_reconcile_error` to the
  dashboard.
- **Silent flip detection**: pre-bar snapshot of `prev_direction`/
  `prev_tier`/`prev_entry_price`. After `on_state`, if the engine still
  thinks it's in position but any of those changed, log a `SILENT_FLIP`
  event (was the source of yesterday's silent churn).
- `_shutdown()` runs `numba.cuda.close()` in a background thread with
  a 5-second timeout. After all cleanup, calls `os._exit(0)` as the
  final hammer to guarantee process exit even if a non-daemon thread
  is still alive.
- Manual FLATTEN from the dashboard now `mark_sent('BAY_CLOSE')` after
  send, so the watchdog can timeout it if NT8 never responds.
