### Breakeven Lock Bug + Fix (3 changes in `core/exit_engine.py`)
Root cause: breakeven lock was **completely inert** due to three bugs:

1. **`_get_stop_price()` ignored `pos.stop_loss`**: recomputed from `sl_ticks` every bar,
   so BE lock's modification to `pos.stop_loss` was never read. Fixed: return `pos.stop_loss`.

   Trades exited before the lock could take effect. Fixed: moved to step 5b.

   the threshold was ~5,000 ticks — max MFE was ~1,500. Fixed: new `be_activation_ticks=4`.

### Task 7: LiveEngine Decomposition — Partial, Reframed

**Original spec** (CLAUDE_CODE_LIVEENGINE_DECOMPOSITION.md) proposed extracting 4 methods
into separate modules/classes (BrainLearner, PingPongManager expansion, ExitWatcher expansion,
named handlers). The goal was reducing LiveEngine from 1,785 to ~1,200 lines.

**Why full decomposition was declined:**
The spec assumed the target methods contained "pure logic" that could cleanly separate.
In practice, all 4 methods (`_brain_learn`, `_compute_life_pct`, `_direction_learn`,
`_flip_position`) read 10-15 LiveEngine attributes each. Extracting them into separate
classes would create parameter-passing wrappers — the new class constructors would need
every attribute passed in, and the methods would just forward-call them. This adds a layer
of indirection without reducing complexity. The methods ARE orchestration, and LiveEngine
IS the orchestrator — moving orchestration out of the orchestrator defeats its purpose.

**What was done instead — shared code paths (the real goal):**
The user clarified the actual motivation: trainer and live should use the same code paths
to prevent logic drift. This reframed the task from "reduce LiveEngine lines" to
"unify duplicated logic into shared core modules."

Concrete action: created `record_trade()` in `core/bayesian_brain.py` — a shared function
that constructs TradeOutcome + calls brain.update() + brain.direction_learn(). Previously
this was duplicated 3x: trainer normal exit (line 983), trainer EOD exit (line 1410),
live _brain_learn (line 1304). Each had its own TradeOutcome construction with slightly
different field handling. Now all three call `record_trade()`.

Also fixed: trainer line 1078 called `brain.direction_learn()` without `tick_value` param,
silently defaulting to 0.50 (correct for MNQ by luck, wrong for other instruments).

LiveEngine's `_direction_learn()` renamed to `_log_direction_bias()` — now read-only
(queries bias table for logging, doesn't update it; update happens inside record_trade).

**Bug fixed**: `capture` variable in `_brain_learn()` line 1344 was referenced but
never computed — would crash at runtime. Added `capture = pnl_ticks / mfe_ticks * 100`.
