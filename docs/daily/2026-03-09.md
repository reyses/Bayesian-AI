## Session 2: Research EE + FF, Breakeven Lock Fix

### Analysis EE — Stop Loss Optimization (KILL fixed SL, PROMOTE breakeven)
- Created `scripts/research/analysis_ee_stop_loss_opt.py`
- **Bug found in v1**: oracle_mae/mfe are direction-blind (mfe=upside, mae=downside).
  For SHORT trades (84% of data), adverse = oracle_mfe, not oracle_mae. Fixed in v2.
- **Fixed SL**: KILL — SL fires 0.5% of trades; exit engine handles everything
- **Breakeven acceleration**: 77% of OOS losers had trade_mfe >= 4 ticks before reversing.
  Moving SL to breakeven at 4 ticks MFE → +$7.5K OOS (+34%), +$29K IS (+33%)

### Analysis FF — Conviction Calibration (KILL)
- Created `scripts/research/analysis_ff_conviction_audit.py`
- IS: conviction NOT predictive (p=0.34). OOS: weak signal (p=0.02, tiny effect)
- Recalibrated logistic AUC: IS=0.538, OOS=0.501 — barely above random
- Calibration systematically overconfident at high conviction (claims 75%, delivers 58%)
- Conviction is a binary filter at best, not a calibrated probability

### Breakeven Lock Bug + Fix (3 changes in `core/exit_engine.py`)
Root cause: breakeven lock was **completely inert** due to three bugs:

1. **`_get_stop_price()` ignored `pos.stop_loss`**: recomputed from `sl_ticks` every bar,
   so BE lock's modification to `pos.stop_loss` was never read. Fixed: return `pos.stop_loss`.
2. **Cascade ordering**: BE lock ran at step 7 AFTER envelope/giveback (steps 6/6b).
   Trades exited before the lock could take effect. Fixed: moved to step 5b.
3. **Threshold unreachable**: used `trail_activation_ticks * 0.6`, but with avg tp_ticks=8,600
   the threshold was ~5,000 ticks — max MFE was ~1,500. Fixed: new `be_activation_ticks=4`.
