## EVENING: New project locked — Bayesian Trade Archetypes via N-D Trajectory Clustering

Long topic-by-topic discussion. Protocol locked, build pending. Full spec at
`research/bayesian_archetypes/project.md`.

### Conceptual progression (topic-at-a-time, locked in order)
1. **Clustering strategy** — peel extremes first (highest mfe_velocity seed),
   gather similar trades, remove, repeat. Each peel = a distinctive archetype.
2. **Feature pool** — ALL ~190 V2 features per bar (not just my originally-
   listed ~19 entry state vector). User flagged the gap; corrected.
3. **Seed criterion** — `mfe_velocity` (the trade's own rate of development).
   Always positive, lookahead-OK for offline clustering.
4. **Per-feature 5% matching** — rejected as default because of zero-crossing
   degeneracy (velocity/z-score features near zero collapse cluster bands).
5. **N-D trajectory representation** — user's key insight: a trade isn't a
   point at entry; it's a sequence of N-D feature vectors over time. Each
   trade gets a **PCA line in 190-D space** as its signature (centroid +
   direction). Similarity = perpendicular distance of candidate trajectory
   points to seed's line, in z-scored units.
6. **Hierarchical r-ladder** — coarse-to-fine clustering. Each seed produces
   a TREE of nested clusters at multiple r values (e.g., 0.5σ → 0.25σ →
   0.125σ). Bayesian table holds the full tree per direction.
7. **Trade decay** (free side-effect) — `d(t)` time series = how trajectory
   drifts from cluster's PCA line over the trade's duration. Three patterns:
   stable / decaying / converging. This unifies:
     - entry classifier (which cluster does a bar match)
     - exit signal (when does live trade decay out of archetype)
     - duration prediction (decay rate)
     - Bayesian online posterior update for direction
8. **CUDA** — PyTorch implementation; batched SVD for trade signatures.

### Open questions (deliberately deferred)
1. Decay metric: raw r-distance vs normalized-to-r-level. Decide after IS
   sanity check visualizes decay curves.
2. "Decayed" threshold: hard-cross of r_coarse vs sustained positive slope
   vs both. Decide when building live evaluator.

### Live applicability — flagged
Clustering uses forward trajectories (full info). Live system at entry
doesn't have forward info. Two paths for live use:
- Match on past-N-bar lead-in trajectory, correlate to forward archetype
- Single-bar match at entry + decay-update as bars arrive
Phase 5 (live evaluator) decides.

### Experiment 1: Cluster variance decomposition — L3 captures real structure

User hypothesis: clustering enables generalization within homogeneous groups.

Audit: for the 3,710 trades assigned to a cluster, **76.2% of signed_mfe
variance is explained by cluster membership alone** (vs 33.7% for global
V2-feature regression). Per-cluster σ median = 34% of global σ. 224 of 298
clusters (75%) have σ_within < 50% global. **Hypothesis confirmed.**

But only 47% of trades cluster — the other 53% are too unique to fit any
r=7 archetype. Those have higher variance ($230 std vs $120 for assigned).

### Experiment 2: Can we predict cluster from entry features or lead-in?

User question: "how does the lookback trajectory look and do those lookbacks
converge in an r predictor?"

Built `regret_leadin_trajectories.py` — per-trade PCA signature of the past
60 bars (5 min) BEFORE entry. Used the same global z-score as forward
signatures (parity).

| Approach | Top-1 acc | Dir match | Test R² |
|---|---|---|---|
| Entry features → cluster (LR, 184D → 298 cls) | 40% | 72% | **−0.05** |
| Entry features → cluster (GBM) | 7% | 56% | −0.66 |
| Lead-in nearest-cluster (5min lookback) | 34% | 50% | −0.65 |
| Lead-in LR → cluster | 28% | 70% | −0.04 |
| Lead-in Ridge → signed_mfe direct | — | — | 0.146 |
| **Direct global feature regression k=3** | — | — | **0.337** ← live ceiling |
| ORACLE (perfect cluster knowledge) | — | — | **0.745** |

**Result: L3 cluster R²=0.745 is the theoretical ceiling, but neither entry
features nor 60-bar lead-in can predict cluster membership well enough.**
Cluster-routed predictions go NEGATIVE R² because the 60-72% misclassifications
have wrong-sign mean_signed.

**Direction-only IS meaningful at 70-72%** — clusters capture direction
reliably, just not magnitude. Hybrid (clusters for direction + L2 features
for magnitude) is the natural next test.

### Honest summary

- L3 captures real archetype structure (R²=0.76 with perfect cluster knowledge)
- But the archetypes emerge from forward-trajectory evolution, not entry signals
- Live prediction via L3 currently does NOT beat L2's R²=0.337
- L3 is a research artifact at this design — needs longer lookback, coarser

## Lookback expansion test — does lead-in help direction?

Tested whether the macro setup INTO entry adds signal beyond the entry bar
itself. Three lookbacks: 60-bar (5 min), 240-bar (20 min), 720-bar (60 min).
Each: PCA centroid+direction in 184-D V2 space → concat with V2 entry → LR.

| Variant | Test AUC | Test Brier | Train-test gap |
|---|---|---|---|
| **Baseline (V2 entry only)** | **0.864** | **0.142** | 0.000 |
| + 60-bar (5 min) lead-in | 0.850 | 0.152 | 0.038 |
| + 240-bar (20 min) lead-in | 0.842 | 0.156 | 0.047 |
| + 720-bar (60 min) lead-in | 0.849 | 0.153 | 0.037 |

**Verdict: ALL lead-in lookbacks HURT.** Train AUC always rises (model latches
onto lead-in features), test AUC always falls (those features are noise on
unseen trades). Calibration deteriorates at every lookback. No monotonic
trend — failure is the approach, not the window.

**Why lead-in PCA doesn't help**:
1. V2 entry features (L1–L3 × 5s–1D) already encode the macro setup. The
   4h/1D-layer features at entry carry the same regime info a lead-in PCA
   would extract.
2. PCA signatures are lossy 2-vector summaries of a 240×184 trajectory matrix.
   Most variance is discarded.
3. Unit-direction vectors going into a linear model are geometrically
   incoherent — the model can't use them without per-trade context.

**Architectural implication**: drop lead-in PCA for direction. Next AUC lever
is non-linear (GBM/CNN), not more features. L3 clusters may still help for
magnitude / risk / exit but NOT for direction routing.

### Bucket finding
MAE is mostly $0 for daisy trades (oracle picks BEST extreme → 84%
MONOTONIC). The 4-class trajectory bucket is too imbalanced; class-weighted
loss forces rare-class predictions and both LR and LSTM end up BELOW the
always-predict-T0 baseline (83.5%). Recommendation: collapse to binary
CLEAN/PULLBACK or drop the trajectory head entirely.

### Architectural conclusion
Drop lead-in trajectory as a model input. The ceiling is in V2 feature
engineering, not model class. Next levers:
- Richer entry features (new TF combos, intra-bar microstructure,
  calendar/event encoding)
- Different target (regret-on-skip vs direction)
- GBM on entry features as a quick non-linear probe for speed/duration
  (where weak non-linear interactions might exist)

### Critical bug found + fixed
V2 layered data has sub-5s irregular timestamps. Dataset builder's
`iloc[::12]` didn't align with ticker's `is_1m_close = (ts % 60 == 0)`.
90% of cache lookups failed. Fix: filter `ts % 60 == 0`.

### Result after fix: ALL 60 CONFIGS LOSE MONEY
Best: T_timing=0.90, T_dir=0.75, TP=$20/SL=$10 → -$14.80/day NET, CI [-$23, -$6].
CI strictly negative — statistically a loser, not noise.

### Killer insight
**Direction classifier is anti-predictive at high-timing-confidence bars.**
- TP/(TP+SL) on combined strategy = 40/57 = 41% (below 50% random baseline)
- Direction classifier 81% acc was on AVERAGE oracle bars (smooth setups)
- High-timing bars are extremes/inflections — direction genuinely uncertain there
- The two classifiers are anti-correlated where it matters most

### Verdict
Two-V2-feature-classifiers stacked DOES NOT WORK as a deployable strategy.
Adding timing filter HURT vs direction-classifier-alone (-$15 vs +$2.54).

### Next: Path A — classifier as filter on existing tier strategies
Tier strategies (FADE_CALM +$13.50/day OOS) fire at price-action moments,
NOT at extremes. Direction classifier should retain 81% accuracy on
tier-fire bars. The two signals likely INDEPENDENT for tier bars.
