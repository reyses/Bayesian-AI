# 2026-05-21 — Zigzag Cold-Start Divergence: Bounded Transient

## Context
The user reframed the per-day zigzag: production
(`build_zigzag_pivot_dataset.py`) runs `detect_swings` on each day's full 5s
closes, anchored at the **day's first bar** — "a beginning-of-time-dependent
feature, and the worst kind." The concern: a live engine launched mid-session
cold-starts its zigzag at a mid-day anchor; a forward pass anchors at the
day's first bar. They will diverge. **The question: do they diverge enough to
be significant?**

This also resolves the Phase-2 question from the zigzag-only refactor — the
original Phase-2 plan ("prime the zigzag with prior-day history") was already
flagged wrong (the forward pass runs per-day, so prior-day priming would
DIVERGE from it). This study measures what the real cold-start gap is.

## Test — tools/zigzag_coldstart_divergence.py (NEW)
Per day, per cold-start offset K ∈ {10%, 25%, 50%, 75%} into the day:
- reference  = `detect_swings(closes[0:])`  — beginning-of-time
- coldstart  = `detect_swings(closes[K:])`  — anchored at bar K

Measured: resync lag (bars from K until the two pivot sequences become an
identical tail — `detect_swings` is deterministic, so once they share a pivot
+ matching suffix they stay identical forever after), transient mis-tiled
legs, and leg-amplitude $ divergence over the transient. Production params
(ATR(14)×4, min_bars=36). **54 days, 216 cold-start samples.**

## Result — BOUNDED TRANSIENT, not significant
| Metric | Result |
|---|---|
| Never-resync | **0 / 216 (0%)** — every cold-start re-converges |
| Resync lag | median 27 min, mean 44 min CI [38, 50], p90 ~70–129 min, max 269 min |
| Mis-tiled legs in transient | **median 0**, mean 0.7, max 4 |
| Mis-tiled leg-amplitude ($ proxy) | mean $103, median $80, CI [$91, $114], max $514 |

Per-offset resync median: 10%→39 min, 25%→22, 50%→20, 75%→47. None ever fail
to resync. **The $103 is a proxy** — a dollar measure of how differently the
two zigzags tile price during the transient, NOT a realized P&L loss.

## Interpretation
The zigzag is **not** a beginning-of-time-dependent feature — it is a
**bounded-lookback feature with ~27 min median memory**. The 0% never-resync
rate proves there is no permanent path-dependence: the running extreme is
memoryless beyond the last pivot, so two runs collapse to identical the
instant they catch a common pivot. A mid-session live launch diverges from
the forward pass for ~27 min median (median 0 divergent legs — the transient
is mostly warmup dead-air), then is byte-identical. Not "back to the drawing
board."

## Phase 2 reframed (docs/JULES_ZIGZAG_ONLY_REFACTOR.md updated)
- STEP 1 (diagnose): the forward pass runs `detect_swings` PER-DAY anchored at
  the day's first bar. **Prior-day priming is REJECTED** — it would make the
  live engine diverge from the validated forward pass.
- STEP 2 (quantify): this study — bounded transient, not significant.
- STEP 3 (fix): `prime_zigzag_from_history` = **same-day catch-up**: at
  session start replay today's elapsed 5s bars (day's first bar → launch)
  through the zigzag. Re-creates the exact per-day anchor → byte-identical
  from bar 1. PRIORITY: low — cheap, makes parity exact, does not gate SIM.

# Evening (autonomous) — ATR-Multiplier Sweep: FLAT pass is oracle-contaminated

## Context
User wants a hot-swappable ATR-multiplier solution and, watching the live SIM,
suspects ATR×4 is too coarse (prior: 2–3). Built `tools/atr_multiplier_sweep.py`
— a FLAT 1-contract zigzag forward pass across ATR multipliers 1–10 (0.25 steps
through 1–4, integers 5–10), IS + OOS, reusing the production Stage A + Stage B
functions. Self-test PASSED: X=4 reproduces the published baseline exactly
($690.10/day IS, $454.02/day OOS).

## Result — ARTIFACT, not a tradeable edge
OOS $/day rises MONOTONICALLY as ATR shrinks — no interior peak across 19 points:

    ATR×10   −$167/day   DayWR 32.7%   TWR −0.21
    ATR×4    +$454/day   DayWR 72.5%   TWR +0.36   <- baseline
    ATR×3    +$833/day   DayWR 75.0%   TWR +0.63
    ATR×2   +$1751/day   DayWR 92.3%   TWR +1.46
    ATR×1   +$3480/day   DayWR 100.0%  TWR +4.25

X=1 = $3,480/day OOS, 100% winning days (51/51), profit factor 5.25 — 7.7× the
trusted baseline with every OOS day green. Not a causal result.

## Why — the FLAT pass trades a hindsight-clean partition
The FLAT pass trades the OFFLINE zigzag. Offline-zigzag legs are a SEQUENTIAL
PARTITION of the price path (cf. the mid-leg finding: 99.8% of consecutive-leg
gaps = 0) — every leg is a genuine pivot-to-pivot swing, ZERO whipsaws by
construction. A causal engine flips at confirmations, many of which are false,
and whipsaws. As the ATR multiplier (→ r_price) shrinks, the offline pass finds
ever more clean swings while the causal whipsaw cost it never pays explodes.
The metric scales with zigzag SUBDIVISION, not tradeability — hence monotonic
and unbounded. X=4 is ~trustworthy only because at a coarse threshold
offline ≈ streaming (the validated ~98% match).

## Verdict
The FLAT hardened-leg sweep CANNOT select the ATR multiplier — valid only at a
fixed X (as a baseline and the denominator of fixed-X B-stack deltas). A real
ATR comparison needs a CAUSAL streaming forward pass that pays the whipsaw
cost. The script's auto-verdict ("pick X=1") is INVALID — annotated in the
report and the script docstring.

## Re-aim (user, same evening): "the goal is lift the OOS bad days"
User granted full NN-training permission + unlimited research latitude, and
stated the real objective: lift the OOS bad days. The ATR sweep was a means;
the goal is bad-day mitigation. Pivoted to a DMAIC run on `research/oos_bad_days/`
— MEASURE the OOS bad days in the validated L5 stack, ANALYZE what separates
them, run one validated experiment (IS-select / OOS-confirm, CIs). ATR/pivot
retrain re-enters as a candidate lever, evaluated causally. Stage-2 B9-retrain
at the FLAT sweep's picks CANCELLED (would retrain at X=1, on a contaminated
metric).

## ANALYZE — do bad days cluster? (`tools/oos_bad_day_autocorr.py`)
Daily P&L lag-1 autocorrelation: IS +0.061 (perm p=0.24), OOS −0.007 (p=0.96)

into bad-day clustering, and B10 already exploits the vol regime.

(not sig), intraday session-stop (sig loss), prior-day-P&L / clustering (iid),

the per-trade drawdown stop (76% recover). The ONE thing that works is the
deployed B-stack: **+$175/day on bad days, significant**. The bad days are the
B-stack's main job and it is doing it; the residual is the irreducible cost of
a trend zigzag meeting choppy markets, and chop is not predictable ahead of
time at the day level.

# Conditional probability table — diagnose-not-trade (entries 1-4)

## Reframe (user)
"I don't want to catch more — I want to be able to **diagnose** that." Not a

probability estimates. A local-LLM (Llama) layer to chain the if-this-then-

## atr_consensus_measure.py — cross-ATR consensus inverts under causality
Offline cross-ATR-level consensus showed a +$57/leg P&L spread. The CAUSAL
version INVERTS it: corr +0.34 -> -0.13. Live, multi-scale zigzag agreement
is a weak leg-MATURITY signal (consensus builds as a leg matures -> high
consensus = late leg = worse entry), not an edge. Same offline-vs-causal
trap family as the FLAT ATR sweep.

## The table — 4 entries (reports/findings/oos_bad_days/)
**Entry 1 — chop begets chop** (`conditional_probability_table.py`): given K
consecutive low-range legs, P(next leg range). After 3 low-range legs P(next
low) = 65% IS vs 33% base; P(breakout) = 8%. Replicates OOS. Chop is sticky.

**Entry 2 — early chop -> longer leg** (`leg_chop_survival.py`): v1 CONFOUNDED
(chop ratio co-grew with leg length -> tautology). v2 fixed-early-window
(chop over the first K tight legs only). Early-choppy wide legs run ~2x
longer — P(C>=K+5): clean 27% -> choppy 48% IS at K=3. Replicates OOS.

**Entry 3 — does the trend survive a chop/fakeout** (`trend_continuation.py`):
v1 CONFOUNDED — the chop table alternated 65/37/64/36/65 by K parity (a
directional measure on leg-anchored windows is eaten by zigzag alternation).
v2 leg-DECOUPLED — trend = sign of the regression slope over a 90-min window,
the zigzag marks the event only; no-K-parity validation gate PASSED. Finding:
WEAK — after a chop/fakeout P(preceding trend continues) ~53-56% vs ~50-52%
base (~+4pp).

**Entry 4 — is the move expended** (`leg_age_hazard.py`, this session):
conditional-survival curve — given a leg has run M min, P(it continues
+X min). **HUMP-SHAPED hazard**: the leg-death rate climbs 1% -> 9%/min over
the first ~5 min, peaks at the ~5-min danger window, then DECLINES — a leg
that clears ~5 min becomes MORE persistent. NOT monotone exhaustion. All four
shape arms significant IS+OOS (IS early +14pp, late +5pp; OOS +10pp, +4pp).
Caught a 5th artifact: my first verdict compared endpoints only (0.5 vs
18 min) and mislabeled the hump "exhaustion" — fixed to a shape-aware
two-arm verdict. min_bars caveat: the steep early arm is partly the
min_bars=36 (3-min minimum zigzag leg) construction floor — above the floor
continuation RISES with age (the above-floor arm is -4pp, sig, IS+OOS). A
pure "old leg = cut it" time-stop is unsupported; only a danger-window-aware
check is backed.

## worst_trades_gallery.py — the worst-trade signature
The N largest losing legs charted in context. Worst trades = fast sharp
adverse spikes ("entry disasters"), not slow bleeds. The extreme tail leans
SHORT but regresses with a wider window (worst-10 = 8 short; worst-100 =
57/43) — regime-conditional (OOS is an up-market), not a structural bias. (I
overstated "8/9 short" from n=9 and corrected it.)

## On "B1-7 would fix it — the target is direction"
User read the worst trades as: a long trend, then chop, then a wrong-way
short because the last leg flipped — a direction problem B1-7 catches. Pushed
back: entry 3 measured exactly that — after a chop/fakeout the preceding
trend continues only ~53-56% vs ~50-52% base. The mechanical counter-trend
entry IS slightly wrong, but a ~+4pp directional edge is too weak for a
B1-7 direction gate to fix profitably — and B1-B6 augmentation was already
tested at -$76/day (the B9 GBM extracts pivot structure from raw V2; stacked
B-preds add noise — the lead-in-PCA failure pattern). Direction is not the
lever here.

## Method lesson — confounds in zigzag conditional tables
Five artifacts caught and corrected this arc. The discipline: (1) never
measure DIRECTION on leg-anchored windows — zigzag alternation parity eats
it; measure leg-decoupled (regression slope over a long window). (2) Chop
must be measured over a FIXED window, not one that co-grows with the
outcome. (3) Read the FULL curve shape, not endpoints. (4) OOS-confirm every
cell. (5) FLAT oracle-zigzag passes are hindsight partitions — not valid
cross-parameter.

## engine_v2 `--account` flag + prompt (code change)
User is about to run NT8 Market Replay (playback) and asked the engine to ask
for / accept the account. `live/launcher.py` already had `--account`, but the
L5 entry point `live/engine_v2.py` did NOT — its `main()` built the config as
`dataclasses.replace(LiveConfig(), pivot_source=...)`, so the account was
silently the `Sim101` LiveConfig default. NT8 playback must trade
`Playback101`.

Changed (`live/engine_v2.py`, ~45 LOC, all in/above `main()` — engine
internals untouched, `py_compile` clean):
- New `--account` argument (default `None`).
- New `_resolve_account(args)`: explicit `--account` wins; `--mock` uses the
  default (the mock bridge has no real NT8); `--headless` requires `--account`
  (cannot prompt); otherwise prompts interactively, listing Sim101 / Sim102 /
  Playback101 and forcing a y/N confirm on an unrecognised account.
- `KNOWN_SIM_ACCOUNTS = ('Sim101','Sim102','Playback101')`.
- The resolved account is passed into the frozen-config rebuild and logged.
- The NT8 `CONNECTED` handler now warns `ACCOUNT MISMATCH` if the bridge
  reports a different account than the engine was configured for — re-instates
  a guard the legacy engine had (`_ALLOWED_SIM_ACCOUNTS`) and the v2 rewrite
  dropped.
- Module docstring usage block updated.

Risk: low — confined to `main()` + helpers; engine decision code untouched.
Blast radius: a non-mock launch without `--account` now prompts instead of
silently defaulting (intended). Rollback: revert the five edits.

(or `python -m live.engine_v2` and answer the prompt).
