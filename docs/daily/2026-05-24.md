# 2026-05-24

## 1. What Changed
**Phase 2: VizEngine Architecture Migration**
The standalone visualization tools in `tools/viz/` were heavily fragmented and crashed when loading massive datasets. We migrated them to a unified plugin architecture powered by `VizEngine`.

- **Dynamic Infinite Scrolling**: `VizEngine` now dynamically loads, concatenates, and caches adjacent trading days as the user pans or zooms out.
- **Unified Core Engine**: All UI logic, geometry persistence, Tkinter event bindings, crosshairs, and bug fixes (e.g., layout snapping via 20px resize) are centralized in `core/engine.py`.
- **Plugin System**: Tools implement the `VizPlugin` interface. They are decoupled from data-loading and event loops, simply drawing overlays via `draw(ax, ax_ind)`.
- **Secondary Indicator Panels**: `VizEngine` conditionally supports a secondary `ax_ind` panel for plugins that require complex feature/ML visualization (via `requires_indicator_panel = True`).
- **Ribbon Sliders & Buttons**: Implemented an extensible on-screen ribbon connected to the Matplotlib toolbar. Plugins can inject custom buttons via `get_buttons()` and native Tkinter Scales via `get_sliders()`.
- **Ported Plugins**:
  - `trade_visualizer` & `swing_inspector`
  - `auto_seeds_generator` & `manual_peak_marker` (replacing old standalone scripts)
  - `feature_marker` (supports 5s timeframe and price/indicator overlays)
  - `classifier_inspector` (sliders moved to the ribbon)

### New file: `training/run_strategy.py`
User: *"in order to run a strategy we need to build a run wrapper that uses the forward pass from core and then passes the data to the strategy, from there it builds csv or parquet with the trades it produced then uses the prob table to analyze the distribution."*

Built. ~200 lines.

**Pipeline**:
```
core_v2/FPS/forward_pass_system.py  ──► BarState stream (V2 cache mode)
                                              │
                                              ▼
training/strategies/zigzag.py ──► ZigzagStrategy.evaluate(state) ──► EntrySignal
                                              │
                                              ▼
                            Single-position tracker (reverse-on-flip)
                                              │
                                              ▼
                       Trades CSV (hardened-legs schema, 10 cols)
                                              │
                                              ▼
                      (optional --analyze) tools/suites/trade_outcome_suite
```

**Output schema** (matches `causal_flat_zigzag_legs_*.csv` consumed by trade_outcome_suite):
`day, entry_ts, leg_dir, entry_price, exit_ts, exit_price, pnl_pts, pnl_usd, r_price, atr_pts`

**Per-day flow**:
1. Read 1m parquet, compute `ATR(14)` (median of last 42 TRs, same convention as `build_zigzag_pivot_dataset.py`).
2. Set strategy `min_reversal_ticks = round(ATR_pts / TICK_SIZE × atr_mult)`.
3. Iterate `ForwardPassSystem(day)` — yields BarState per 5s.
4. On `EntrySignal` (flip): close current position at 5s close, open new opposite at same price (zero slippage, $2 round-trip commission).
5. Day-end: force-close any open at the last bar's 5s close.

**Strategy bug caught + fixed**: `ZigzagStrategy.evaluate()` was reading `state.price` (the 1m close held constant across 12 5s bars per FPS code line 178) — defeated the 5s-precision requirement. Now reads `state.ohlcv_5s['close']`. Without this fix, the strategy was effectively running at 1m granularity.

## 13. Probability-table analysis promoted into --analyze flow

User: *"I think we need to promote to the analyze the prob tables."* Wired the trade_outcome_suite as the canonical post-strategy analysis target.

### What changed
- `tools/suites/trade_outcome_suite/trade_outcome_suite/excursions.py`: added `'strategy_run'` entry to the `SOURCES` registry. Points at the runner's canonical IS/OOS output paths (`reports/findings/strategy_runs/zigzag_{is,oos}_atr4.csv`) and the corresponding 5s OHLCV directories.
- `training/run_strategy.py` `--analyze`: now invokes `run_all.py --source strategy_run --rebuild` (was `--source causal_flat`, which read the wrong files). Includes a safety check: if the user passed a custom `--out` that doesn't equal the canonical path, the runner prints a warning and skips analyze rather than silently analyzing the wrong file.

### Note on the smoke verdicts
The 15 verdict lines that printed during smoke (e.g. *"Median MFE $46/$76 (IS/OOS); P(close>0) 28%/46%"*) are computed on a sample of N=2 days total. They are PIPELINE SMOKE, NOT a baseline. Full runs need the complete IS+OOS day list; verdict numbers will change substantially with sample size.

### Why this matters going forward
Every future strategy that gets registered in `training/run_strategy.py` (B-filtered zigzag, new tier classes, etc.) automatically inherits the prob-table analysis. Once the canonical CSV at `reports/findings/strategy_runs/<strategy>_{is,oos}_atr4.csv` is fresh, `--analyze` produces the 15-question diagnostic in one step. No per-strategy plumbing needed.

### Expected Impact on Metrics
**Zero** — wiring only. No new code path, no behavior change to existing baselines. Runner output CSV format unchanged.
