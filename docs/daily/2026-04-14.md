## Critical Discovery: 100% Feature Parity

The original engine_v2 used `compute_79d_from_aggregator` (per-bar SFE)
which diverged from build_dataset's `process_one_day` (batch SFE per TF).
**7.5% exact match — 92% of features wrong**.

Root cause: SFE called once-per-day shares state across the window;
called once-per-bar starts fresh each time. Same bars, different states.

Fix: `LiveFeatureEngine` (NEW) loads pre-built TF parquets at startup,
appends new live bars, runs batch SFE per TF using the same windowing
as `process_one_day`. **Result: 100.0% parity** (130,943/130,949 cells
exact, 85/91 features perfect, 6 acceleration diffs at bar 0 only —
fixed by loading prev_velocities from checkpoint).

## Engine V2 Hardening

**Connection safety (3 layers of defense):**
1. CONNECTION_LOST from bridge → `_broker_connected = False` → block orders
2. Stale bars (>60s no arrival) → assume NT8 panic → block orders
3. Catch-up flood (avg inter-arrival < 1s) → process features but skip trading

All use `time.monotonic()` — timezone/clock independent (user is on
Pacific time, original logic used wall clock comparisons).

**State sync:**
- Periodic POSITION request every 12 bars (~60s)
- If NT8 reports flat but engine in-trade → `engine.force_close()`
- Bridge sends ORDER_REJECTED on null _account / submit failure
- Engine reads ACCOUNT_UPDATE messages → dashboard shows actual NT8 PnL

**Memory cleanup on shutdown:**
- `numba.cuda.close()` releases GPU
- `lfe = None`, `engine = None`, `gc.collect()` releases RAM

- X button now closes within 1s (was 30s timeout)

## Gravity Well Concept (user insight)

z_high / z_low are not just lines — they're **attractor wells**. Price gets
pulled back toward regression mean from extreme z. Deeper historical extreme =
stronger gravity.

Implications:
- Long near 1h floor = trading WITH gravity (good)
- Long near 1h ceiling = trading AGAINST gravity (bad)
- Headroom in your direction matters more than absolute price

`training/level_tracker.py` (NEW) computes z extremes over rolling windows
(1D = 60 days persistent, 1h = 5 days tactical). Used as entry filter:
block if min headroom in direction < 0.5 z.
