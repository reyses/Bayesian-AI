### ⚠️ CRITICAL DISCOVERY — selection bias in probability table

Tested the "strongest cell" (P_revert=0.71, z_1h_low ≤ -3σ AND slope_15m ≤ -0.5):

```
Naive entry at FIRST bar where condition true:
  IS:   46.9% win rate (predicted 71%)   $-14.4/day
  Val:  48.5% win rate                    $-3.7/day
  OOS:  46.3% win rate                    $-4.8/day

CUSP-ON-Z entry (z just started rising after sustained <-3σ):
  IS:   58.6% win rate                   +$2.3/day  ★
  Val:  50.0% win rate                    $-3.0/day
  OOS:  35.9% win rate                    $-8.8/day  ← OOS regime mismatch
```

**The 0.71 P_revert is misleading.** The probability table averages across ALL
bars where the condition holds, including bars in the MIDDLE of a crash run
where the bounce hasn't happened yet AND bars near the actual cusp where it
will. Firing at the FIRST condition-bar gets you mostly mid-crash entries,
which crash further before bouncing.

**The cusp-on-z transition (z stopped falling) recovers ~12pp of win rate** but
still has IS-only edge — OOS regime differs.

This explains why the user's manual picks work (visual hindsight identifies
the literal cusp bar) but mechanical rules using the same anchor conditions
fail (they fire across all bars, not just the cusp bar).
