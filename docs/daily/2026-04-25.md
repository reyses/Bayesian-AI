### `tools/cascade_pivot_quality.py` — pivot-conditioned quality analyzer

Built today. For every 1m zigzag pivot at R, looks up `<TF>_z_se` from
`DATA/ATLAS/FEATURES_5s/` (no SFE recompute — the features are already there
since 2026-04-17 lookahead-fix regen). Counts cascade alignment IN the pivot's
trade direction. Walks forward to next opposite pivot, records MFE/MAE/ETD/
capture/leg-P&L. Buckets pivots by alignment count AND by continuous
`dir_energy` deciles (sum of |z_se| signed in pivot direction).

Run on full ATLAS, 345 days, 11K pivots at R=30 / 43K at R=10.

**Result:** cascade alignment is real but moderate.
- Top decile of `dir_energy` shows WR +4-5pp and MFE +20-25% vs bottom decile.
- Confirms IS+OOS at R=10 (4-pp gap holds OOS).
- Effect rare in extreme form (≥4 TFs aligned at |z|>1.5 = ~0.5% of pivots).
- **MFE rises but ETD rises with it.** Capture % stuck around −100% to −180%
  across all alignment buckets. Cascade does NOT fix the give-back problem —
  that's still a trail-stop job.

**Cascade and trail are orthogonal improvements** that target different failure
modes (entry quality vs exit capture).

## ADDENDUM (later in session) — Regime + Envelope FILTER is the v1.5 lever

User refined the cascade concept:

> "if a 1h pivot fires that tells us the overall direction, and if we do z
>  bands along that direction it should tell us the expected noises, and if
>  we couple that noises with the zigzag triggers of the lower bands it
>  should tell us the expected width of the 1m bands so it's easier for us
>  to catch fakeouts and estimate the minimax"

Built `tools/regime_envelope_quality.py`. For each 1m pivot, compute:
- 1h regime (WITH or AGAINST) from latest 1h zigzag pivot at confirm time
- Envelope band from `|1h_z_se|`: inside / near / edge / outside / extreme

Result on full ATLAS (11K pivots):
- **Worst bucket OOS: AGAINST-regime + extreme** (\|z\|>2σ): WR 30%, mean
  −$12.28/trade across 209 trades. The "fakeout-at-exhaustion" trap.
- **Best bucket IS: WITH-regime + outside** (1.5-2σ): WR 42%, mean +$4.54
  across 480 trades.
- **Noise-fakeout bucket: WITH-regime + inside** (\|z\|<0.5σ): mean −$3 to −$4.
  These are 1m flips happening within normal 1h noise band — they reverse back.

**OOS gain is 3.3× the IS gain** — opposite of overfit. The filter captures

**The regime+envelope filter is specifically a R_1m=30+ phenomenon.** At R=10
the 1m trades are too short (~9 min mean) to be influenced by 1h structure.
At R=30 (mean leg ~30 min) the 1h regime/envelope has time to matter.

This is an important calibration finding. For NT8 deploy, MUST use R=30.

## Change report — DynamicRiskManager refactor of v1.2 (in-place, NOT deployed yet)

**What changed**

Refactored the inline two-phase trail + hard-SL logic in
`docs/NT8_ZigzagRunner_v1.2.cs` into a dedicated `DynamicRiskManager` class
(state machine: `Null → Initial → Tier1 → Tier2`). Strategy now drives the
manager via three calls:

- `riskMgr.OnInitialFill(fillPx, dir, fillPx)` from OnExecutionUpdate on
  every fresh fill — places hard SL at fill ± 25pt using the actual fill
  price (gap-safe).
- `riskMgr.EvaluateStopState(Position, Close[0])` from OnBarUpdate per
  bar close — drives state transitions, ratchets stop tighter when
  favorable.
- `riskMgr.ResetState()` from OnExecutionUpdate on flat fill.

Manager invokes a `RouteStopOrder(stopPrice, direction, currentPrice)`
callback the strategy supplies. Callback validates the stop against
currentPrice and either calls `SetStopLoss` (valid) or fires a market
exit with reason `TrailMissedBreach{Long|Short}` (invalid =
trail-breach-missed scenario).

**What files**

- `docs/NT8_ZigzagRunner_v1.2.cs` — only file touched.
  - Removed: orphan trail-state fields (`trailArmed`, `trailWaterMark`,
    `trailLastStopPx`) and all references in Configure / OnExecutionUpdate
    (3 branches) / OnBarUpdate.
  - Added: `private DynamicRiskManager riskMgr` field, `RouteStopOrder`
    callback method, top-level `StopState` enum and `DynamicRiskManager`
    class (in same NinjaTrader.NinjaScript.Strategies namespace).
  - Banner updated; legacy CHANGELOG entries preserved as historical
    record.

**Why this matters / behavior parity**

Manager constants are POINTS-based and parameterized by the strategy's
existing NinjaScriptProperty inputs:

- `INITIAL_STOP` ← `HardStopLossPoints` (default 25pt = $50)
- `T1_ACTIVATION` ← `TrailActivatePoints` (default 10pt = $20)
- `T1_TRAIL` ← `TrailDistancePoints` (default 5pt = $10)
- `T2_ACTIVATION` ← derived as `TrailDistancePoints / TrailPercent`
  (default 5/0.10 = 50pt = $100, matches existing crossover)
- `T2_TRAIL_PCT` ← `TrailPercent` (default 0.10)

So default behavior is bit-for-bit equivalent to the prior v1.2 inline
logic. Manager is unit-testable in isolation (no NT8 callbacks
referenced from inside the class — only via the supplied delegate).

**What to look for in the next NT8 run**

1. **Compile clean** — F5 in NinjaScript Editor should produce 0 errors,
   0 warnings. Two new types in the file (StopState enum, DynamicRiskManager
   class) are declared in the same namespace as the strategy class.
2. **Trades fire** — fresh fill should immediately show a hard-SL line
   on the chart at fill_price − 25pt (long) / fill_price + 25pt (short).
3. **Trail ratchets** — once unrealized profit ≥ $20 (Tier1), stop
   should jump to HWM − 5pt and visibly ratchet up each bar as HWM
   advances. At unrealized > $100 (Tier2), trail tightens to HWM − 10%
   of peak profit (90% lock).
4. **Missed-breach exits** — if a 1m bar gap pushes price past the
   computed stop level, expect a market-exit row with
   `exit_reason = TrailMissedBreachLong/Short` instead of NT8 silently
   rejecting the modify.
5. **CSV ledger** — unchanged columns; mfe_pts / mae_pts / capture_pct
   should still populate.

**Expected impact on metrics**

Zero net change vs prior v1.2 (parity refactor). Validation gate is
"Playback for one session reproduces v1.2 trade timing within 1 bar of
prior version's output". If not, manager has a bug.

**Deploy gate**

## Change report — StagnationMonitor integration into v1.2 (in-place, NOT deployed yet)

**What changed**

User pasted a `StagnationMonitor` class. Integrated verbatim as a third
top-level class in `docs/NT8_ZigzagRunner_v1.2.cs`, alongside the
existing `DynamicRiskManager` and the strategy class. Re-introduces the
negative-bars timeout that was removed during v1.2 release prep, but
with structural cleanup:

- Hardcoded `MAX_NEGATIVE_BARS = 5` (compile-time const) instead of the
  prior runtime-configurable `NegativeBarsTimeout = 10`. To retune, edit
  the constant in the class body.
- Class-internal `lastEvaluatedBar` guard prevents intra-bar
  multi-increments (defensive against being called twice per bar).
- Class-internal `ResetState` is private and triggered on Flat detection
  inside `RequiresFlatten`. Strategy recreates the instance on flip /
  fresh-entry fills (since the flip path never reads as Flat) — keeps
  the class API surface as the user supplied it.
- Strategy queries `stagnationMon.RequiresFlatten(Position, Close[0],
  CurrentBar)` in OnBarUpdate AFTER `riskMgr.EvaluateStopState`. On
  true, fires `ExitLong(Position.Quantity, "StagnationExitLong", "")` /
  `ExitShort(...)` and returns from OnBarUpdate.

**What files**

- `docs/NT8_ZigzagRunner_v1.2.cs` — only file touched.
  - Added: `private StagnationMonitor stagnationMon` field; instance
    creation in Configure; instance recreation in OnExecutionUpdate flip
    + fresh-entry branches; RequiresFlatten check + market-exit in
    OnBarUpdate; `StagnationMonitor` class definition (verbatim from
    user paste) inside the namespace after `DynamicRiskManager`.
  - Removed: my speculative `NegativeBarsTimeout` NinjaScriptProperty
    (was a placeholder before the user's actual paste arrived).
  - CHANGELOG block at top of file describes both the
    DynamicRiskManager refactor (prior change) and the StagnationMonitor
    addition.

**Pushback noted**

The previous negative-bars timeout (v1.2.3 lineage, default = 10) was
removed because it was "firing too aggressively / breaking entry
flow." Reintroducing the same feature class is a regression risk.
Mitigations baked into this design:

1. The hardcoded threshold is now 5, not 10 — TIGHTER. If the prior
   bug was over-eager firing, this version fires faster, not slower.
   This deserves Playback validation before live trade.
2. `lastEvaluatedBar` guard rules out the "called twice per bar →
   double-increment → premature trigger" failure mode.
3. Strategy-agnostic class — easier to unit-test in isolation.
4. The market-exit happens AFTER `riskMgr.EvaluateStopState`, so any
   stop adjustment for the bar still lands first. Order: trail-stop
   update → stagnation check → (return).

**What to look for in the next NT8 Playback run**

1. Compile clean (F5 in NinjaScript Editor).
2. Trades that sit underwater for 5 consecutive 1m bars without ever
   touching breakeven should fire a market exit with reason
   `StagnationExitLong` or `StagnationExitShort` in the CSV ledger.
3. Trades that briefly turn positive (even by $0.01) should reset the
   counter — verify by inspecting CSV: long-held winners should NOT
   carry stagnation exits.
4. Validate that the prior failure mode (entries silently failing, no
   trades firing at all) does NOT reproduce. If it does, comment out
   the RequiresFlatten block in OnBarUpdate as the kill switch.

Unknown — the behavior change vs prior v1.2 is a NEW exit reason.
First-order prediction: cuts a small number of "stuck" losing trades
earlier than the hard SL, capping per-trade loss tighter than $50 in
those scenarios. Could either help (lower MAE on losers) or hurt
(prematurely cutting trades that would have recovered to breakeven via
mean reversion). Needs A/B Playback against current v1.2 to measure.
