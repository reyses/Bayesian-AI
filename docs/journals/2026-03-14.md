## Pipeline Architecture
```
Multi-TF ZigZag (7 TFs) + human seeds
  -> Extract lookback (10 bars) + segment waveforms
  -> Normalize (zero at entry, unit range) + resample to 32 pts
  -> UMAP (32D -> 2D)
  -> HDBSCAN in 2D -> natural shape families
  -> 16D centroid per primitive (physics-based)
  -> checkpoints/shape_primitives.pkl
  -> Phase 2 K-Means uses primitives as init centroids
```

## Session 3: 90% Confidence + Magnitude-Aware Clustering

Key insight from user: "V and v are not the same" — a 500t V_REVERSAL and a 30t v_reversal
should be SEPARATE primitives. Shape recognition, not MFE prediction.

### Changes to `tools/shape_primitive_builder.py`
- **Confidence 99% → 90%**: `CONFIDENCE_Z = 1.645` (was 2.576). With 50% tolerance + CV=1:
  n_required drops from ~27 to ~11, allowing more fine-grained primitives
- **Magnitude features in UMAP**: Waveform vector now 34D (32 shape + 2 magnitude)
  - `log1p(abs_change_ticks) / 6.0 * weight` — overall move size
  - `log1p(max_range_ticks) / 6.0 * weight` — waveform range
  - Default weight = 3.0 (meaningful but not dominant vs 32 shape dims)
  - `--magnitude-weight` CLI flag to tune (0 = shape only, like before)
- **New CLI flags**: `--confidence` (z-score override), `--magnitude-weight`
- **centroid_waveform**: Stores only 32-point shape (no magnitude), since downstream
  K-Means init needs pure shape. Magnitude varies per member within a primitive.
- **R2 computation**: Only on shape dimensions (first 32), not magnitude

### Rationale
- 90% confidence = "these shapes are real with 90% probability" — appropriate for shape
  recognition where the exit engine handles all risk management
- Magnitude as UMAP features: big V and small v naturally separate in 2D embedding
  because their magnitude dimensions differ, even though their shape curves are identical

## Session 4: Two-Stage Primitive Architecture

Key architectural insight from user: "the primitive is the output, the input is the primitive
lookback... once we are in trade then is when we provide the shape of the segment primitive so
the giveback knows what to expect."

Rewrite from single-pipeline to **two-stage architecture**:

### Entry Primitives (lookback-based)
- Input: 10-bar lookback geometry (6D) + 192D multi-TF context at entry bar
- Purpose: "What setup is this?" -> inform entry decisions
- Trend continuations naturally cluster together
- Per-TF clustering with UMAP 198D->2D + HDBSCAN
- Output: `checkpoints/entry_primitives.pkl`

### Exit Primitives (segment-based)
- Input: 32-point normalized segment shape + 2 magnitude features (34D)
- Purpose: "What move to expect?" -> calibrate giveback exit thresholds
- Per-primitive calibration: giveback_pct, delay_bars, envelope_halflife_mult, peak_bar
- V_REVERSAL = tight/fast giveback, RAMP = loose/slow giveback
- Output: `checkpoints/exit_primitives.pkl`

### Files Created
- `core/shape_primitives.py` (~230 lines) — all data structures:
  - `EntryPrimitive`, `ExitPrimitive` dataclasses
  - `EntryPrimitiveLibrary` with `match()` and `get_centroids_for_tf()` methods
  - `ExitPrimitiveLibrary` with `match_partial()` and `get_exit_params()` methods
  - `extract_lookback_geometry()` — 6D: slope, curvature, efficiency, range, end_position, monotonicity

### Files Modified
- `tools/shape_primitive_builder.py` — complete rewrite (~900 lines):
  - Split into `--entry`, `--exit`, `--all` modes
  - Entry path: `compute_entry_features()` (6D geometry + 192D context per seed)
  - Exit path: `normalize_segments()` (segment-only, not lookback+segment)
  - `calibrate_exit_params()` — data-derived giveback/envelope params per cluster
  - `build_entry_primitives_from_clusters()` / `build_exit_primitives_from_clusters()`
  - Shared: ZigZag detection, quality filter, per-TF clustering loop, bootstrap
  - `GEOMETRY_WEIGHT = 5.0` (prevents 6D drowning in 192D context)
  - `UMAP_ENTRY_DEFAULTS` with n_neighbors=50 (higher for 198D input)

### Design Decisions
- Lookback = last 10 closes before entry (not the previous segment itself)
- 192D context = snapshot of all 12 TFs' 16D features at entry timestamp
- Exit calibration is DATA-DERIVED from member outcomes, not hand-tuned per shape
- Graceful fallback everywhere: missing primitives = current behavior unchanged

## Session 5: Builder Output Quality Fixes

First run on ATLAS_1WEEK produced 68 entry + 75 exit primitives. Two quality issues found:

### Bug 1: Entry R2=0.000 (all primitives)
- **Root cause**: R2 formula used `centroid = mean(feats, axis=0)` for both ss_res and ss_tot.
  Since centroid IS the column mean, `ss_res == ss_tot` always → R2 = 1 - 1 = 0.
- **Fix**: Compute R2 on unweighted 6D geometry only, using scalar mean for ss_tot
  (matches the exit R2 pattern which correctly gives 0.3-0.8).

### Bug 2: Exit giveback_pct=20% floor (all primitives)
- **Root cause**: Calibration used `net_change / total_range` (segment efficiency).
  ZigZag segments are clean swings between pivots → efficiency ≈ 1.0 always →
  `1.0 - 1.0 * 0.8 = 0.20` → hits floor for every primitive.
- **Fix**: Derive giveback from **peak position** + **monotonicity** instead of efficiency.
  Early peak (V-shape) → tight giveback (0.30). Late peak (ramp) → loose (0.80).
  Monotonicity modulates ±0.05. Gives meaningful differentiation between shape families.

## Session 7: Shape-Aware Exits via --shapes Flag

Same pattern as `--lookback`: opt-in CLI flag, no separate pipeline.

### Architecture
Instead of a separate UMAP+HDBSCAN shape primitive pipeline, calibrate exit params
per-template from member oracle outcomes using the existing K-Means infrastructure.

After template creation, for each template:
1. Examine member PatternEvents' `window_data` (oracle forward window)
2. Compute peak position, monotonicity → giveback_pct
3. Classify dominant shape (IMPULSE/V_REVERSAL/RAMP/OTHER)
4. Derive delay_bars, envelope_hl_mult
5. Store on template → flows into pattern_library

At trade time: template's shape params → PositionState → giveback + envelope

### Priority chain for shape_params in giveback:
1. Exit primitives (from `--primitives`, if matched with confidence ≥ 0.3)
2. Template shape calibration (from `--shapes`)
3. Tier-based thresholds (default, always available)

### Files Modified
- `training/trainer.py` — `--shapes` flag, calibration call after template creation,
  shape fields in pattern_library dict
- `core/fractal_clustering.py` — `PatternTemplate` shape fields,
  `calibrate_template_shapes()` method
- `core/exit_engine.py` — `template_shape_params` on PositionState,
  `open_position()` extracts from lib_entry, `evaluate()` fallback chain

### Usage
```bash
# Baseline
python training/trainer.py --fresh

# Shape-aware exits only
python training/trainer.py --fresh --shapes

python training/trainer.py --fresh --lookback --shapes
```

## Session 10: Trailing Stop + V-Reversal Exit + SHORT Bias Discovery

### Problem: Breakeven Lock Wastes Profit
Min-hold experiment (IS: $53,293, 5,050 trades, 97.0% WR) revealed:
- 1,940 SL exits averaged $0/trade — breakeven lock snaps to entry+1t then stops
- 1,481 trades gave back 90-100% of peak profit at $1/trade avg
- Breakeven lock is one-time: price can run 20+ ticks, come all the way back, exit at $0

### Solution: Direction-Aware Trailing Stop
Replaced one-time breakeven snap with continuous trailing stop in `core/exits/breakeven.py`:
- Activation: MFE >= 4 ticks (same as before)
- Trail: continuously ratchet SL at `trail_pct * MFE` behind peak favorable price
- **Direction-aware**: LONG trail_pct=0.65 (tighter), SHORT trail_pct=0.50 (looser)
- Rationale: LONG cycle = gradual rise → sharp drop (need tighter trail to catch fast giveback)
  SHORT cycle = sharp drop → gradual recovery (can afford looser trail)

### V-Reversal Exit (4-bar rule)
Added to `core/exit_engine.py` cascade (step 9b, after trailing stop):
- Track `bars_since_peak` counter on PositionState (resets on new peak, increments otherwise)
- If `bars_since_peak >= 4` AND `breakeven_locked` AND `pnl > 0`: exit immediately
- Logic: 8-min cycle confirmed from human seeds → at 15s bars, 4 bars off peak = reversal confirmed
- Active during min-hold period — V-reversals don't need to wait for hold to expire

### SHORT Bias Discovery
User identified asymmetric bias: LONG 1,594 trades (59.6% correct dir) vs SHORT 3,375 trades (50.9% correct).
System takes 2x more shorts despite longs being more accurate.

**Root cause**: Market microstructure asymmetry. Drops are faster/sharper than rallies → all sensors
(velocity, DMI, z-score, bands) respond more strongly to fast moves → natural SHORT bias in detection.

Direction-aware trailing stop partially addresses this — longs get tighter protection.

- `core/exits/breakeven.py` — complete rewrite: one-time snap → continuous direction-aware trail
- `core/exit_engine.py` — `bars_since_peak` field, MFE tracking update, V-reversal exit (step 9b)
