PURPOSE
-------
Build a standalone screening tool that identifies tradeable patterns
directly from raw price data — no training checkpoints, no templates.pkl,
no dependency on the training pipeline.

The goal: find WHERE in price/time the tradeable setups live, THEN layer
physics features on top for context. Not the other way around.

THE PRICE I-MR METHOD
---------------------

What is I-MR?
  Statistical Process Control (SPC) chart. Used in manufacturing to
  detect when a process goes out of control. We apply it to price.

I Chart (Individual):
  - Each data point = one 15m close price
  - Center line = mean(close) from warmup period
  - UCL_I = center + 2.66 * MR_bar  (E2 constant for n=2)
  - LCL_I = center - 2.66 * MR_bar
  - Points outside limits = price is in "special cause" territory

MR Chart (Moving Range) — SIGNED:
  - MR = close[t] - close[t-1]  (signed, not absolute)
  - Positive = price went up, Negative = price went down
  - Every sign change and magnitude change is visible
  - MR_bar = mean(|MR|) from warmup period
  - UCL_MR = 3.267 * MR_bar  (D4 constant for n=2 subgroup)
  - |MR| > UCL = regime transition (price behavior changed character)

Why signed MR matters:
  - Every change in the MR is a potential pattern
  - Sign flips show direction reversals
  - Magnitude shows intensity
  - Runs above/below zero show trend persistence
  - UCL breaks in either direction mark regime boundaries


REGIME DETECTION
----------------

How regimes are found:
  1. Calibrate control limits from warmup period (first N days)
  2. Walk through analysis window
  3. New regime starts when |MR| > UCL_MR
  4. Merge tiny regimes (< 8 bars) into neighbors
  5. Each regime gets: direction (price trend), mean price, volatility

What a regime represents:
  - A period where price behavior has a consistent character
  - Bounded by special cause events (big bar-to-bar jumps)
  - The natural segmentation of price action
  - Replaces the fake ADX quartile bins

Regime metadata:
  - regime_id: sequential integer
  - start_idx, end_idx: bar range
  - n_bars: duration
  - mean_price: average close in regime
  - volatility: mean |MR| within regime
  - direction: LONG if price trended up, SHORT if down
  - price_change: close[-1] - close[0] within regime


ORACLE MFE/MAE
--------------

For each analysis bar:
  - Look ahead 16 bars (4 hours at 15m)
  - max_up = max(future highs) - entry price
  - max_down = entry price - min(future lows)
  - Direction from REGIME (not z-score):
    LONG regime -> MFE = max_up, MAE = max_down
    SHORT regime -> MFE = max_down, MAE = max_up
  - Win = MFE > MAE

THE THREE QUESTIONS METHOD
--------------------------

The screening follows a structured sequence of three independent analyses.
Each question builds on the answer to the previous one.

ANALYSIS A: Can we explain the PRICE?
  Y = close[t], X = 193 fractal context features
  Method: correlation screening + stepwise OLS regression
  Answer (1-day): adj-R2 = 0.9519 (YES, 95.2%)
  Key factors: 30m ADX (r=0.90), 30m DMI_diff (r=0.88), 15m ADX (r=0.87)
  Conclusion: The fractal context describes WHERE price is at any moment.

ANALYSIS B: Can we explain the DIRECTION standalone?
  Y = sign(close[t+1] - close[t]), X = 193 features (+ price anchor)
  Method: same correlation screening + stepwise OLS
  Answer (1-day): adj-R2 = 0.0870 (NO, only 8.7%)
  Key finding: all top factors have NEGATIVE correlation (mean reversion)
  Conclusion: Direction cannot be explained independently. The model has no
  anchor — it doesn't know WHERE price is, so it can't determine which way
  it goes. Adding raw price as a feature didn't help because the relationship
  between price level and direction is non-linear.

ANALYSIS C: Can we derive direction FROM the price model?
  Insight: if we can predict price[t] and price[t+1] with 95% accuracy,
  then direction = sign(predicted[t+1] - predicted[t]).
  Method: use the fitted price model from A, predict consecutive bars,
  derive direction from the difference.
  Risk: residual noise (~25 pts) may overwhelm typical moves (~9 pts).
  This tests whether 95% price R2 is "good enough" for directional use.

WHY THIS SEQUENCE MATTERS:
  - A establishes what the context CAN explain (price level)
  - B proves direction needs an anchor (can't be explained in isolation)
  - C tests whether the price model IS the anchor (derived direction)
  - Each analysis stands alone — no cross-referencing, no unified signals
  - The order is important: you must answer A before B makes sense,
    and B must fail before C is motivated

ANALYSIS C RESULTS (1-day, 2026-02-28)
  Overall accuracy: 55.7% (49/88) — barely above chance
  SNR: 0.668 (poor — residual std=21.8 > typical move=14.6)

  BUT — confidence-gated accuracy:
    |predicted delta| > 0:   55.7% (88 samples)
    |predicted delta| > 5:   68.8% (64 samples)
    |predicted delta| > 10:  73.9% (46 samples)
    |predicted delta| > 20:  73.9% (23 samples)

  KEY FINDING: The price model DOES carry directional information, but only
  when it predicts a move larger than the noise floor. When predicted delta
  exceeds ~10 points, accuracy jumps to 74%. Below that, noise dominates.

  This is a NATURAL CONFIDENCE GATE — the model itself tells you when to
  trust its direction (large predicted delta) and when to sit out (small
  predicted delta). You don't need a separate direction model; you need
  a confidence threshold on the price model's consecutive predictions.

  Asymmetry: DOWN prediction (60.5%) more accurate than UP (52.0%).
  Consistent with Analysis B's mean-reversion pattern — the model is
  better at calling reversals from extended states than continuations.

CONCLUSIONS SO FAR
------------------

1. Price explanation WORKS at all scales (89-97% R²) but the useful
   information shifts to higher TFs as the window grows.

2. Direction standalone is DEAD at all scales (3-9% R²). This is not a
   data problem or a feature problem — direction is not linearly
   predictable from spatial context.

3. Derived direction from price model has a SWEET SPOT at ~1 week:
   - Too short (1 day): not enough samples, noisy (56% overall)
   - Sweet spot (1 week): 67% overall, 80% at >20 pt confidence gate
   - Degrading (1 month): 60% overall, 68% at >20 pt gate
   - Dead (1 year): 50% — pure coin flip, confidence gate fails

4. The confidence gate is real but ONLY works within the sweet spot.
   It does not rescue the model at scales where SNR < ~0.2.

5. A single global linear model cannot handle multi-regime data.
   The path forward must involve either:
   a) Rolling windows (fit model per ~1-week window, slide forward)
   b) Regime-conditional models (fit per regime, derive direction locally)
   c) Non-linear methods (trees, neural nets that partition input space)
   d) Relative features (not absolute price, but price-relative-to-regime)


CRITICAL BUG: LOOK-AHEAD BIAS IN HIGHER TF ALIGNMENT (2026-02-28)
------------------------------------------------------------------

DISCOVERY: All previous results had look-ahead bias.

The problem:
  When sitting at a 15m bar with open timestamp t=10:00, the code looked up
  the higher TF bar with searchsorted(timestamps, t, side='right') - 1.
  This finds the bar whose OPEN time <= t. But OHLCV timestamps are bar OPEN
  times, so the bar at t=10:00 on a 1h chart covers 10:00-11:00 and its
  close/high/low/volume aren't known until 11:00.

  Look-ahead per TF (for base 15m bar):
    30m bar at 10:00 → completes at 10:30 → 15 min look-ahead
    1h  bar at 10:00 → completes at 11:00 → 45 min look-ahead
    4h  bar at 8:00  → completes at 12:00 → up to 3h45m look-ahead
    1D  bar at open  → completes at close → up to ~23h look-ahead
    1W  bar at open  → completes Friday   → up to ~5 days look-ahead

  TFs SMALLER than base (5m, 3m, 2m, 1m, 30s, 15s) are fine — their bars
  complete within the 15m base bar window.

THE FIX:
  For TFs with bar duration > base TF duration: use N-1 (previous completed bar).
    searchsorted(timestamps, t, side='right') - 2  (instead of - 1)
  For TFs with bar duration <= base TF: keep - 1 (bar completes within base bar).

  This means for 15m base at 10:00:
    30m → uses bar at 9:30 (covers 9:30-10:00, COMPLETED)
    1h  → uses bar at 9:00 (covers 9:00-10:00, COMPLETED)
    4h  → uses bar at 4:00 (covers 4:00-8:00, COMPLETED)
    1D  → uses previous day's bar (COMPLETED)
    1W  → uses previous week's bar (COMPLETED)

  Applied to both alignment locations:
    - build_stacked_matrices() line ~286 (full pipeline)
    - Step 8 fractal context line ~1791 (Three Questions analysis)

IMPLICATION:
  All prior R² results are INVALID — they were inflated by future information.
  The 95% price R² was partly fake: higher TF features contained future price data.
  Must re-run all analyses to get honest numbers.

ANALYSIS D: RATE-OF-CHANGE / PATTERN RECOGNITION (2026-02-28)
--------------------------------------------------------------

QUESTION: Does adding temporal information (feature deltas between
consecutive bars) improve price or direction explanation?

METHOD:
  For each bar at time t, compute delta = features[t] - features[t-1].
  This gives 192 rate-of-change features capturing HOW features are
  changing, not just their current level. Test three things:
    D1: Does level+delta improve price R2?
    D2: Does level+delta improve direction R2?
    D3: Does the combined price model give better derived direction?

RESULTS (1-day, 88 samples, N-1 fix active):

  D1 - PRICE:
    Level only:    adj-R2 = 0.9401
    Level + delta: adj-R2 = 0.9401  (no improvement)
    Delta features in price model: 0/20
    -> Delta features carry NO additional price information.
       The spatial snapshot already fully captures price level.

  D2 - DIRECTION:
    Level only:    adj-R2 = 0.0735 (7.4%)
    Level + delta: adj-R2 = 0.2049 (20.5%)  *** +13.1% ***
    Delta features in direction model: 9/20 (almost half!)
    -> RATE-OF-CHANGE DRAMATICALLY HELPS DIRECTION.
       Direction R2 nearly tripled from 7.4% to 20.5%.

  D3 - DERIVED DIRECTION:
    Level only:    67.0%
    Level + delta: 67.0%  (no improvement)
    -> The delta features don't help the PRICE model, so derived
       direction from price (Analysis C) is unchanged. But they
       help DIRECT direction modeling (Analysis B -> D2).

  TOP DELTA FACTORS FOR DIRECTION:
    dt_d4_30m__self_dmi_diff     r=-0.268  (DMI diff CHANGING at 30m)
    dt_d5_15m__z_score           r=+0.244  (z-score CHANGING at 15m)
    dt_d6_5m__self_adx           r=-0.241  (ADX CHANGING at 5m)
    dt_d5_15m__self_pid          r=+0.228  (PID CHANGING at 15m)
    dt_d4_30m__z_score           r=-0.219  (z-score CHANGING at 30m)

  INTERPRETATION:
    The direction signal lives in the TRAJECTORY of features, not their
    level. When DMI diff is FALLING at 30m (r=-0.27) or z-score is
    RISING at 15m (r=+0.24), these predict the next bar's direction.
    This is exactly what a spatial snapshot misses: the "momentum of
    the indicators" rather than the indicators themselves.

  CAVEAT: This is 1-day only (88 samples). Must validate at 1-week
  and 1-month to see if the 20.5% direction R2 holds up.

  The 1-day "tripling" (7% -> 20%) was real signal amplified by small
  sample size. The underlying effect is ~4-5% incremental at best.

  BOTTOM LINE: Pattern recognition (rate-of-change) is NOT the answer
  to the direction problem. The direction signal is fundamentally weak
  in linear models regardless of features used.


ANALYSIS E: dP/dT-GROUPED SIGNAL AMPLIFICATION (2026-02-28)
------------------------------------------------------------

INSIGHT: "small samples that are similar are easier to explain the direction"
  The signal isn't weak -- it's DILUTED. Within a single regime, features
  carry real directional information. Mix 259 regimes and signals cancel.
  GROUP by signed dP/dT to keep similar bars together.

METHOD:
  Split analysis bars into 3 groups by signed MR (DOWN / FLAT / UP terciles).
  Run full Three Questions (price R2, direction R2, derived direction)
  independently per group. Compare per-group vs global results.

ANALYSIS G: LAPLACIAN SUB-SEGMENTATION (d2p/dt2)
==================================================

Date: 2026-02-28
Method: Compute discrete Laplacian (curvature) of price:
  d2p/dt2[t] = close[t+1] - 2*close[t] + close[t-1]
Within each I-MR regime, create sub-segments where curvature sign
flips (convex<->concave) or curvature MR exceeds UCL. This adds the
ACCELERATION layer that I-MR (velocity) misses.

PHYSICS ANALOGY:
  I-chart    = position (where price is)
  MR-chart   = velocity (how fast it moves)
  Laplacian  = acceleration (is it speeding up, slowing down, reversing?)

1-DAY RESULTS:
  I-MR regimes: 2 -> Laplacian sub-segments: 20 (10x more granular)
  Avg sub-segment size: 4.6 bars

  COHERENCE COMPARISON:
  I-MR regimes   -> avg endpoint std: 53.1 ticks (2 regimes)
  Laplacian subs -> avg endpoint std: 25.6 ticks (6 qualifying sub-segs)
  Improvement: +51.7% TIGHTER

  Best sub-segments:
  S12 (R1, CONCAVE): 6 shapes, std=21.3, WR=100%
  S14 (R1, CONCAVE): 5 shapes, std=26.1, WR=100%
  S2  (R0, CONVEX):  5 shapes, std=5.2,  WR=80%

INTERPRETATION:
The Laplacian sub-segmentation cuts endpoint noise IN HALF compared
to I-MR alone. Sub-segments based on curvature sign (convex vs concave)
capture distinct price SHAPES, not just time periods. This is the
bridge from "group by when" to "group by shape."

REFERENCE: Shi 2018 "Signal Pattern Recognition Based on Fractal
Features and Machine Learning" — 5 fractal dimensions (Box, Katz,
Higuchi, Petrosian, Sevcik) + Random Forest -> 96% signal
classification. Our curvature is analogous to their fractal dimension:
a shape descriptor that distinguishes waveform types.

SHAPE DETECTION METHODS (from DSP literature):
- Dynamic Time Warping (DTW): aligns similar shapes at different speeds
- Fourier Descriptors: shape fingerprint, scale/rotation invariant
- Template matching via cross-correlation: match against known library
- Ref: dsp.stackexchange.com/questions/64722/typical-shape-detection-algorithms

scales. The goal: at 1-year scale, build a segment LIBRARY where
similar curvature profiles predict direction, then enrich with the
